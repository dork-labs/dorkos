# 235. Site-Local Drizzle Schema for Marketplace Telemetry (Not packages/db)

## Status

Accepted (extracted from spec: marketplace-04-web-and-registry)

## Context

ADR-0234 established Neon Postgres + Drizzle ORM as the single source of truth for marketplace install telemetry, hosted in `apps/site`. A natural follow-up question: where does the Drizzle schema live?

DorkOS already has a `packages/db` package that uses Drizzle for the local SQLite database (agent registry, sessions, activity log). On the surface, putting the new Postgres schema for `apps/site` in the same package looks like consolidation. In practice, the two databases serve fundamentally different domains:

| Aspect          | `packages/db` (existing)              | `apps/site` telemetry (new)         |
| --------------- | ------------------------------------- | ----------------------------------- |
| Engine          | SQLite (better-sqlite3)               | Postgres (Neon serverless HTTP)     |
| Driver          | `drizzle-orm/better-sqlite3`          | `drizzle-orm/neon-http`             |
| Runtime         | Node (server process)                 | Vercel Edge (Edge Functions)        |
| Domain          | Local agent fleet, sessions, activity | Public install analytics            |
| Trust boundary  | Single-user local data                | Public, multi-tenant aggregate data |
| Migration cycle | Tied to server release cadence        | Tied to site deploy cadence         |

Sharing one package would mean forcing every consumer of `@dorkos/db` (the Express server, the CLI, test utilities) to pull in `@neondatabase/serverless` and Postgres-specific Drizzle chunks they will never use. It would also couple two release cadences that have no reason to move together.

## Decision

Keep the marketplace telemetry Drizzle schema **site-local** to `apps/site`:

- Schema lives at `apps/site/src/db/schema.ts`.
- The `db` instance lives at `apps/site/src/db/client.ts` (instantiates `drizzle(neon(env.DATABASE_URL))`).
- The Drizzle config lives at `apps/site/drizzle.config.ts`.
- Migrations live at `apps/site/drizzle/` (generated by `drizzle-kit generate`, applied by `pnpm db:migrate`).
- Both `db:generate` and `db:migrate` are scripts in `apps/site/package.json`, runnable from anywhere via `pnpm --filter @dorkos/site db:generate` or directly inside `apps/site` with `pnpm db:generate`.
- The schema is **not** exported from `apps/site` to anything else. It is consumed only by Server Components in `apps/site/src/app/marketplace/` and by the `/api/telemetry/install` Edge Function.
- `packages/db` continues to use SQLite via `drizzle-orm/better-sqlite3` exclusively, with no Postgres dependencies.

If a future feature ever needs to share schemas between the local server database and the public site database (we cannot currently identify a use case), the migration path is to extract the shared types into a third package (`packages/db-shared` or similar). That extraction is a small refactor and is explicitly not pre-built today.

## Consequences

### Positive

- **`packages/db` stays SQLite-only.** Server, CLI, and test consumers never pull in `@neondatabase/serverless`, Postgres-specific Drizzle chunks, or Edge runtime polyfills they don't use. Bundle size and dependency surface stay tight.
- **Co-located migrations.** `apps/site/drizzle/` lives next to the code that uses it. A site engineer running `pnpm db:generate` doesn't need to context-switch into a sibling package.
- **Independent release cadence.** The site can ship schema migrations on its own deploy cycle without coordinating with a server release.
- **Zero cross-package coupling.** The local-server data model and the public-site telemetry model can evolve independently. A breaking change to one cannot break the other.
- **Matches existing app-local data conventions.** `apps/server` already owns its own data layer through `packages/db`; mirroring that with `apps/site` owning its own Postgres schema keeps the "each app owns its persistence" mental model intact.

### Negative

- **No shared package if a future feature needs cross-domain types.** If we ever want, say, the local server to know about public install counts (to render badges in the in-product Dork Hub), it would need to fetch those counts via HTTP rather than import a shared schema. We accept this — the indirection is minor and the use case is hypothetical.
- **Two `drizzle.config.ts` files in the monorepo.** Slight duplication of Drizzle boilerplate. Acceptable for the clarity gain.
- **Site engineers learn Postgres + Neon driver in addition to whatever else they touch.** This is a one-time cost and is documented in `contributing/marketplace-telemetry.md`.

## Alternatives Considered

- **Put the Postgres schema in `packages/db`.** Rejected. Forces every consumer of `@dorkos/db` to pull in Postgres dependencies they don't use, couples release cadences that should stay independent, and mixes two engines (SQLite and Postgres) inside one package without a clear ownership story.
- **Create `packages/db-postgres` for the public-site Postgres schema.** Rejected as premature. There is exactly one consumer (`apps/site`) and exactly one schema file. Extracting a package today buys nothing and adds workspace overhead. If a second consumer ever appears, the extraction is straightforward.
- **Inline the schema next to each query call site (no central `db.ts`).** Rejected. Even one schema file deserves a single source of truth — having `marketplace_install_events` defined in two places is the exact bug Drizzle exists to prevent.
