## Issues Found

### 1. Ghost Message / Silent Delivery Failure in Relay Mode — Bug (P1)
**Observed:** After Message 4 completed, sending Message 5 ("What is 2+2?") appeared to succeed in the UI — a user bubble and assistant response appeared. However, the JSONL was never updated (still 32 lines), cost didn't change ($0.03), and the "response" was a replay of Message 4's content. The user bubble contained concatenated text from both Message 4 and Message 5.

**Expected:** Message 5 should be delivered to the SDK session via Relay, appear in the JSONL as a new user message, and produce a fresh assistant response ("2 + 2 = 4").

**Root cause:** Race condition in the Relay subscribe-first handshake. The SSE `/stream` endpoint sends `stream_ready` after relay subscription is active. When messages are sent in quick succession, the new message's POST may fire before the new SSE subscription is established, or the previous subscription's sync_update may be misinterpreted as the new message's streaming response. Documented in `research/20260306_sse_relay_delivery_race_conditions.md`.

**Key files:**
- `apps/client/src/layers/features/chat/model/use-chat-session.ts` — `waitForStreamReady()` (lines 17-34), relay EventSource effect (lines 244-292), message submit (lines 355-360)
- `apps/server/src/services/runtimes/claude-code/session-broadcaster.ts` — `registerClient()` sends `stream_ready` (lines 129-130)
- `apps/server/src/routes/sessions.ts` — Relay POST handler (lines 250-262)

**Recommendation:** Ensure the subscribe-first handshake is atomic per-message — the new SSE subscription must be confirmed (`stream_ready` received) before the POST fires. Consider adding a message-specific correlation ID so the client can distinguish sync_updates from actual streaming responses.

### 2. Model Name Display Inconsistency — UX Issue (P3)
**Observed:** During live session, status bar shows friendly name "Haiku 4.5". After history reload, it shows the full model ID `claude-haiku-4-5-20251001`.

**Expected:** Consistent display — either always friendly name or always full ID.

**Root cause:** Live session uses client-side model mapping (dropdown label), while history reload reads the raw model ID from session metadata/JSONL.

**Recommendation:** Normalize model display in the status bar to always use the friendly name mapping regardless of data source.

### 3. Scroll Position Not Restored on History Load — UX Issue (P3)
**Observed:** After hard refresh, the page loads scrolled somewhere in the middle of the conversation rather than at the bottom (most recent message).

**Expected:** On history load, scroll to the bottom to show the latest message.

**Recommendation:** Add scroll-to-bottom on initial history load completion.
