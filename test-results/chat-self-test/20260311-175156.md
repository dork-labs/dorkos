## Issues Found

### Issue 1: React Duplicate Key Storm — Bug

**Observed:** 74 duplicate key errors fire on page load before any interaction. ~17-24 per keystroke, ~160 per streaming response. Console is completely polluted during normal use.

**Expected:** Zero "Encountered two children with the same key" warnings.

**Root cause:** `AssistantMessageContent.tsx:121` uses index-based keys for text message parts:
```tsx
// apps/client/src/layers/features/chat/ui/message/AssistantMessageContent.tsx:121
<div key={`text-${i}`} className="msg-assistant">
```
When the `parts` array is restructured during streaming (e.g., when a `text_delta` event adds or modifies parts), React reconciles the new array against the old. If parts shift positions or are added mid-array, two siblings can momentarily share the same `text-${i}` key. This fires for every `text_delta` SSE event during streaming (~17 errors per event at the observed rate).

By contrast, tool call parts correctly use stable unique IDs (`key={part.toolCallId}`), which don't collide.

**ADR context:** ADR-0005 (Zustand + TanStack Query state). The streaming state is managed via refs (`currentPartsRef`) and `setMessages`, which triggers React reconciliation on every text_delta. The key instability is in the reconciliation, not the state model.

**Recommendation:** Assign stable UUIDs to text parts when they're created in `stream-event-handler.ts`, then use those IDs as React keys:
```tsx
// In createStreamEventHandler, when creating/appending text parts:
currentPartsRef.current = [...parts, { type: 'text', text, id: crypto.randomUUID() }];
// In AssistantMessageContent:
<div key={part.id ?? `text-${i}`} className="msg-assistant">
```
The 74 initial-load errors may have a separate root cause (possibly session list keys in the sidebar) that warrants a follow-up investigation.

### Issue 2: Empty Session ID Causes API 400/404 Errors — Bug

**Observed:** `GET /api/sessions//tasks` → 400, `PATCH /api/sessions/` → 404 on every new session before first message is sent. Model and permission mode changes silently fail.

**Expected:** These queries should not fire when session ID is null/empty. Model and permission mode changes should either be deferred or stored locally until a session is created.

**Root cause:** `ChatPanel.tsx` coerces a null session ID to an empty string and passes it to hooks that immediately fire API calls:

```tsx
// apps/client/src/layers/features/chat/ui/ChatPanel.tsx:37
const taskState = useTaskState(sessionId ?? '');  // '' instead of null
// ChatPanel.tsx:114
const { permissionMode } = useSessionStatus(sessionId ?? '', ...);  // same

// apps/client/src/layers/features/chat/model/use-task-state.ts:48-53
const { data: initialTasks } = useQuery({
  queryKey: ['tasks', sessionId, selectedCwd],
  queryFn: () => transport.getTasks(sessionId, selectedCwd ?? undefined),
  staleTime: 30_000,
  refetchOnWindowFocus: false,
  // ← NO enabled: !!sessionId guard
});

// apps/client/src/layers/entities/session/model/use-session-status.ts:52-56
const { data: session } = useQuery({
  queryKey: ['session', sessionId, selectedCwd],
  queryFn: () => transport.getSession(sessionId, selectedCwd ?? undefined),
  staleTime: 30_000,
  // ← NO enabled guard
});
```

The `useChatSession` hook correctly guards its history query:
```tsx
// use-chat-session.ts:181-193 — CORRECT pattern to follow
enabled: sessionId !== null,
```

**Recommendation:**
1. Add `enabled: !!sessionId` to `useTaskState`'s query (line 48)
2. Add `enabled: !!sessionId` to `useSessionStatus`'s query (line 52)
3. Optionally: change `ChatPanel` to pass `sessionId` (possibly null) directly, rather than coercing to `''`

### Issue 3: Orphaned User Message Bubble on Relay Failure — Bug

**Observed:** Message 3 user bubble ("Write a minimal HTML page...") appeared in the UI but had no corresponding JSONL entry. The bubble persisted until page reload, then disappeared.

**Expected:** User messages should either persist (appear after reload) or never appear (not be added to state before server confirmation). An inconsistent in-between state (visible now, gone on reload) is confusing.

**Root cause:** `use-chat-session.ts:371-379` adds the user message to React state optimistically — before the Relay transport confirms delivery:

```tsx
// use-chat-session.ts:371-379 — OPTIMISTIC ADD
const userMessage: ChatMessage = {
  id: crypto.randomUUID(),
  role: 'user',
  content,
  parts: [{ type: 'text', text: content }],
  timestamp: new Date().toISOString(),
};
setMessages((prev) => [...prev, userMessage]);
// Only then: await transport.sendMessageRelay(...)
```

In Relay mode, the POST returns 202 (accepted) but the message may fail to reach the agent (dropped in transit, agent offline, etc.). The JSONL is only written when the SDK actually processes the message. If the Relay fails silently after the 202, the optimistic bubble has no JSONL counterpart and disappears on reload.

**ADR context:** ADR-0003 (JSONL as single source of truth). By design, "reload shows exactly what's in JSONL." Optimistic UI that doesn't match JSONL is a known tension.

**Recommendation:** Options in order of preference:
1. **Show delivery failure:** If the SSE stream doesn't emit a `done` event within a timeout (e.g., 30s), remove the optimistic message and show an error "Message may not have been delivered."
2. **Confirm before adding:** Wait for the Relay `stream_ready` SSE event before adding the optimistic bubble (the `stream_ready` flow already exists per the Relay spec). If `stream_ready` never arrives, don't add the bubble.
3. **Accept as known behavior:** Document that in Relay mode, a message bubble appearing then disappearing on reload indicates a Relay delivery failure. Low priority if Relay reliability is high in practice.

### Observation: Mid-Stream History Refetch Causes Transient Duplicate User Bubble — UX Issue

**Observed:** During Message 4 (TodoWrite with multi-tool sequence), the "Use TodoWrite..." user bubble appeared twice during live streaming. After page reload, only one bubble appeared.

**Expected:** One user bubble per user message.

**Root cause:** During Relay streaming, SSE `sync_update` events invalidate TanStack Query's messages cache (`queryClient.invalidateQueries(['messages', sessionId])`). This triggers a history refetch mid-stream. If the refetch happens while the JSONL contains a partially-written `tool_result` user message (which is SDK-internal — the tool result feedback sent back to the agent), the transcript parser at `transcript-parser.ts:231-284` may briefly include it as a visible user message before the suppression logic (`hasToolResult → continue`) takes effect.

On reload, the JSONL is complete and the suppression works correctly.

**Recommendation:** In `use-chat-session.ts`, suppress `invalidateQueries` for the messages query while status is `'streaming'`. Instead, queue the invalidation to run after the `done` event fires:
```tsx
// use-chat-session.ts — after done event:
if (type === 'done') {
  setStatus('idle');
  queryClient.invalidateQueries({ queryKey: ['messages', sessionId] });
}
```
This prevents mid-stream history refetches that can surface transient SDK-internal messages.
