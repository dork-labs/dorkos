## Phase 6 — Issue Analysis & Research

### Root Cause: Message 3 Truncation (Relay SSE partial delivery)

**Traced to:** `apps/client/src/layers/features/chat/model/use-chat-session.ts` lines 270-277

The `sync_update` handler on the relay EventSource lacked a streaming guard:

```typescript
// BEFORE FIX (buggy):
eventSource.addEventListener('sync_update', () => {
  queryClient.invalidateQueries({ queryKey: ['messages', sessionId, selectedCwdRef.current] });
  queryClient.invalidateQueries({ queryKey: ['tasks', sessionId, selectedCwdRef.current] });
});
```

When the Skill tool (frontend-design) completed its exchange (turn 1), the JSONL was written and fired `sync_update`. Without the guard, `invalidateQueries` ran mid-stream, triggering a GET /messages refetch. The refetch returned history where the assistant message had a **server-side UUID** (from JSONL), different from the **client-side UUID** in `assistantIdRef.current`. The incremental update appended the history message. Subsequent `text_delta` events called `updateAssistantMessage(clientUUID)` which still found the in-memory message, but competing React state updates from the history refetch left only "Using" visible in the DOM before the refetch clobbered the optimistic state.

**Fix (in uncommitted staged changes):**
```typescript
eventSource.addEventListener('sync_update', () => {
  if (statusRef.current === 'streaming') return;  // ← guard added
  queryClient.invalidateQueries({ queryKey: ['messages', sessionId, selectedCwdRef.current] });
  queryClient.invalidateQueries({ queryKey: ['tasks', sessionId, selectedCwdRef.current] });
});
```

### Root Cause: 503 Storm

**Traced to:** `apps/client/src/layers/features/chat/model/use-chat-session.ts` lines 184-189

The `refetchInterval` callback was polling GET /messages every 3s in Relay mode (same as legacy mode):

```typescript
// BEFORE FIX (buggy):
refetchInterval: () => {
  if (isStreaming) return false;  // No relay guard
  return isTabVisible ? ACTIVE_TAB_REFETCH_MS : BACKGROUND_TAB_REFETCH_MS;
},
```

In Relay mode, polling is redundant — `sync_update` SSE events already drive history invalidation. The 3s polling overwhelmed the Vite proxy connection pool (Vite has limited concurrent connection slots; multiple active sessions × 3s polling = many concurrent requests → 503).

**Fix (in uncommitted staged changes):**
```typescript
refetchInterval: () => {
  if (isStreaming || relayEnabled) return false;  // ← relayEnabled guard added
  return isTabVisible ? ACTIVE_TAB_REFETCH_MS : BACKGROUND_TAB_REFETCH_MS;
},
```
