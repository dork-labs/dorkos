## Phase 6 — Issue Analysis & Research

### Bug 1: 503 Storm on GET /messages in Relay Mode

**Root cause A:** `refetchInterval` in `use-chat-session.ts:184-190` fires every 3s in Relay mode. The callback guards on `isStreaming` (line 185) but NOT on `relayEnabled`. Since `ACTIVE_TAB_REFETCH_MS = 3000` and `MESSAGE_STALE_TIME_MS = 0`, every 3 seconds a GET /messages poll fires even though Relay mode uses `sync_update` SSE events for history invalidation.

**Root cause B:** Express 4 async error handling. The GET /messages route handler (`routes/sessions.ts:116-140`) lacks a `try/catch`. In Express 4, unhandled async rejections don't propagate to error middleware — the request hangs. The Vite proxy (`http-proxy`) times out and returns 503. Confirmed: curl (both direct and via Vite) returns 200 consistently. The browser's persistent SSE connection exhausts proxy connection slots, making subsequent GET requests more likely to hang and timeout.

**Evidence:**
- `curl http://localhost:4241/api/sessions/83839b5f-.../messages` → 200 every time
- Browser network log → 200 on first request, then 503 for all subsequent (~3s interval)
- `ACTIVE_TAB_REFETCH_MS = 3000`, `MESSAGE_STALE_TIME_MS = 0` confirmed in `apps/client/src/layers/shared/lib/constants.ts:56,72`
- 503 storm active on phantom session `c2392cb8` AND correctly-polled session `83839b5f`

**Research:** `research/20260307_relay_streaming_bugs_tanstack_query.md` (full analysis, multiple fix options)
**Spec:** `specs/fix-relay-streaming-bugs/` — already exists with spec, tasks, and task JSON

**Fix (1 line):** Add `|| relayEnabled` to `use-chat-session.ts:184` refetchInterval callback.
**Also fix:** Add `try/catch` with `next(err)` to `routes/sessions.ts:116` GET /messages handler.

### Bug 2: TodoWrite Tool Call Card Spinner Stuck After Streaming

**Observed:** After Message 4 (TodoWrite) stream completed (stop button disappeared, `done` event received), the TodoWrite tool call card showed a **blue `running` spinner** instead of the expected **green checkmark**. Persists until page reload, after which the checkmark displays correctly.

**Expected:** Tool card transitions from `running` → `complete` (green checkmark) when `tool_call_end` event is processed.

**Root cause (hypothesis):** `tool_call_end` is emitted in `sdk-event-mapper.ts:74` (confirmed). In Relay mode, `sync_update` SSE events fire unconditionally during streaming (lines 270-273 in `use-chat-session.ts`), triggering `invalidateQueries`. History refetches during streaming can race with the streaming state. When history is re-seeded from the initial-seed branch (`!historySeededRef.current`), it overwrites `currentPartsRef` state with stale data, potentially losing the `tool_call_end` transition. The `done` event handler (lines 265-284) sets `setStatus('idle')` but does NOT sweep remaining `running` tool calls to `complete`.

**Supporting evidence:** After reload, JSONL has `status: 'complete'` for the TodoWrite tool call — the SDK recorded it correctly. The bug is in live streaming state management only.

**Affected code:**
- `apps/client/src/layers/features/chat/model/stream-event-handler.ts:176-183` (tool_call_end handler)
- `apps/client/src/layers/features/chat/model/use-chat-session.ts:265-285` (done handler)
- `apps/client/src/layers/features/chat/model/use-chat-session.ts:270-273` (sync_update handler — no isStreaming guard)

**Recommendation:** The `isStreamingRef` guard fix for Bug 1's `sync_update` handler (already in `specs/fix-relay-streaming-bugs/`) likely also fixes this spinner bug by preventing the history re-seed race. Should be added as an explicit test case.
