# Sessions

A session is a single conversation with one of your agents. Every message you send, every tool the agent uses, and every response you receive belongs to a session. Sessions are the central unit of interaction in DorkOS. DorkOS runs them on three runtimes (Claude Code, Codex, and OpenCode) and shows them all in one list.

## Session Lifecycle

### Creating a Session

When you start a new conversation, a session is created with:

- A **unique ID** (UUID) that identifies the session across all clients
- A **working directory** that tells the agent where to operate on your filesystem
- A **permission mode** that controls whether tool calls require your approval

You can create sessions through the DorkOS UI, the REST API, or by starting a conversation with the Claude Code CLI directly.

### Sending Messages

Sending a message triggers a turn: `POST /api/sessions/:id/messages` returns `202 Accepted` immediately, and the response streams over the session's durable event stream (`GET /api/sessions/:id/events`). On that stream you receive real-time events:

- **Text deltas** — Incremental chunks of Claude's response
- **Tool calls** — Actions Claude wants to take (reading files, running commands, etc.)
- **Approval requests** — Prompts for you to approve or deny a tool call
- **Task updates** — Progress on multi-step operations

### Session Locking

Only one client can write to a session at a time. When you send a message, DorkOS acquires an exclusive lock on that session using your client ID (sent via the `X-Client-Id` header). If another client tries to write to the same session, it receives a 409 conflict response with the lock holder's ID and acquisition time.

The lock is held for the duration of the turn — it is released when the turn completes or errors, and auto-expires after **5 minutes** as a backstop.

<Callout type="info">
  Multiple clients can view the same session simultaneously — only sending messages requires an
  exclusive lock.
</Callout>

## Storage

DorkOS keeps no separate session database. Each runtime owns its own session storage, and DorkOS reads from every runtime to build one combined list.

- **Claude Code** stores each session as a transcript file on your disk. That file is the single source of truth for Claude Code sessions.
- **Codex** and **OpenCode** keep their own session stores, managed by their own tools.

If DorkOS cannot reach one runtime, it still shows the others and tells you which one it missed.

### Claude Code transcript files

Each Claude Code session has a file at:

```
~/.claude/projects/{project-slug}/{session-id}.jsonl
```

These files hold every interaction as newline-delimited JSON (JSON, one record per line). DorkOS reads them to extract:

- **Session metadata**: ID, title (from the first user message), timestamps, and permission mode
- **Message history**: the full conversation, including your messages, agent responses, and tool results
- **Session list**: built by scanning the project directory for all `.jsonl` files

Metadata is computed on every request, so it always reflects the current state of the file.

### Cross-client visibility

Because Claude Code sessions are plain files on disk, any client that can read the same files sees the same sessions:

- A session started in the **DorkOS web UI** appears in the **Claude Code CLI**
- A session started from the **CLI** shows up when you open DorkOS
- The **Obsidian plugin** reads from the same location

Start a task in one tool, pick it up in another.

## Session Sync

When multiple clients are open, DorkOS keeps them in sync through the session's durable event stream — no polling, no re-fetching.

### How Sync Works

1. Each client subscribes to `GET /api/sessions/:id/events`
2. On connect, the server sends a **snapshot** — completed history, the in-progress turn, session status, and any pending approval prompts
3. From there, every client receives the same live events with monotonic sequence numbers — including turns triggered by other clients or by the CLI
4. On reconnect, the client resumes from its last sequence number (`Last-Event-ID`) and the server replays exactly what it missed; if the gap can't be served, a fresh snapshot is sent instead

A pending approval prompt survives a refresh, a session switch, or a second window: it rides the snapshot, and its countdown resumes where it left off.

### The Session List

The sidebar stays live the same way: a single global stream (`GET /api/events`) announces `session_upserted`, `session_removed`, and `session_status` events as sessions are created, deleted, or change state — across every client and the CLI.

## Session Metadata

Every session shows the same core details in DorkOS: an ID, a title, timestamps, and a permission mode. For Claude Code sessions, DorkOS reads them from the transcript file:

| Field               | Source                                  |
| ------------------- | --------------------------------------- |
| **ID**              | UUID from the JSONL filename            |
| **Title**           | First user message in the transcript    |
| **Preview**         | Beginning of the last assistant message |
| **Created**         | File creation timestamp                 |
| **Updated**         | File modification timestamp             |
| **Permission mode** | Extracted from the SDK init message     |

## Working Directories

Each session operates within a working directory that determines where Claude can read and write files. The working directory is:

- Set when the session is created
- Persisted in the URL as the `?dir=` query parameter (in standalone mode)
- Validated against a configurable boundary to prevent access outside allowed paths

The directory picker in the DorkOS sidebar lets you browse and select a working directory before starting a session.
