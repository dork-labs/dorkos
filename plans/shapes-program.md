# Shapes & BYOA — Program Plan

**Date:** 2026-07-17 · **Status:** draft for Dorian's review
**Basis:** `research/20260717_shapes-byoa-positioning.md` (full evidence trail), `meta/positioning-202607/`, `research/20260708_generative_ui_standards_dorkos.md`

## Goal

Unify generative UI + extensions + marketplace into one product story — **Shapes** (installable, forkable bundles that change what DorkOS is for you) — riding the **BYOA / copilot-flip** strategic spine (apps expose data + tools via MCP; users bring their own agent, model, and UI), **without disturbing the Act-1 launch** already in flight. The program runs decisions → foundations → proof → surfaces → GTM, in that order, because every downstream artifact changes meaning depending on the decisions.

## Phase 2 — Proof (the demo-claim gate for Shapes)

The first shapes are therefore _assembly_, not greenfield — which also proves the two upgrade paths the marketplace story needs: **plugin → shape** (flow) and **extension → shape** (linear-issues).

## Eval cadence (D5 detail)

- **Per-PR**: deterministic tests only (free, already exist). A 3–5-eval smoke tier runs **only when label-triggered** (mirroring the review-label pattern) or when eval-adjacent paths change; cheap model; ~cents.
- **Nightly on main**: full suite, mid-tier model; transcripts saved as JSONL artifacts; failures auto-file tracker items; pass-rate trend tracked, single-run reds retried once then quarantined (flaky list), never silently skipped.
- **Release gate**: full suite green required before `system:release` proceeds — this is the demo-claim gate made executable.
- **Multi-release days** (several releases in one day): the gate is **memoized, not skipped**. Eval results pin to a commit SHA + the set of eval-relevant paths (turbo-style affected mapping, same discipline as `pnpm verify`). A release whose diff since the last green run touches no eval-relevant paths (docs/changelog/site-only) **reuses the green result instantly**; one that does touch them re-runs **only the affected evals**, not the whole suite. First release of the day gets the full run; same-day follow-ups are incremental.
- **Skip escape hatch**: `system:release --skip-evals` exists for hotfix emergencies, with guardrails: a reason is required and recorded in the release's internal notes; the smoke tier still runs (never fully blind); disallowed when the diff touches a marketing-gated pillar's paths; a skipped release leaves a "gate debt" marker that the next nightly run clears or escalates.
- **Weekly deep**: real-provider connector evals (real Gmail/Slack sandboxes), expensive-model runs, long-horizon shape evals.
- **Budget cap** per run enforced in the runner; estimated ~$1–3/full run at mid-tier pricing → nightly ≈ $30–90/mo.

## Explicitly not doing

- No company rename; no "world's first meta-harness" claim (Omnigent, June 2026); "meta-harness" is category/SEO vocabulary only.
- No in-chat paid placement (D6). No taxing community shapes.
- No hosted SaaS at launch (managed data layer is a post-launch, post-evidence line).
- No Act-2 repositioning before D1 triggers; no marketing any shape capability that fails its evals.
- No "video editor" in any pitch (not credible near-term; it burns trust with the beachhead).

## Success criteria

1. One prompt installs, one prompt modifies, one prompt uninstalls a shape — proven by green evals in CI.
2. Reference Linear Ops shape installed by a non-author user in < 2 minutes from the marketplace.
3. Fork→tweak→share loop demonstrated in a < 90-second real-UI capture.
4. Site ships the Shapes pillar page + ≥6 category SEO routes with zero unverified claims.
5. Act-2 trigger dashboard exists (even as a checklist) so the repositioning decision is evidence-driven, not vibes-driven.
