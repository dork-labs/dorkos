# Right-Panel Workbench — File Explorer, Viewers, Browser, Terminal

**Slug:** right-panel-workbench
**Author:** Dorian + Claude (orchestrated ideation; 5 parallel research agents, 2026-07-08)
**Date:** 2026-07-08

---

## 1) Intent & Assumptions

- **Task brief (verbatim scope from Dorian):**
  1. **File Explorer** in the right panel: opens to the session's cwd, worktree-aware, views common file types, add/edit/delete files and folders. Leverage the existing folder explorer if possible ("we already have a folder explorer, but it doesn't currently display files").
  2. **File Viewer**: the agent can open markdown in the canvas today; it should also open images and websites. A Linear ticket exists for opening local HTML/internal websites in the canvas (CORS issue).
  3. **Web pages**: open web pages in the canvas/right panel with light chrome — back/forward, refresh, editable URL.
  4. **CLI**: an embedded terminal opening to the cwd/worktree; explore options including Ghostty Web.
  5. The agent should be able to open files/webpages in the panel; assess whether the two-tab (agent profile + canvas) structure must change; design for extensibility (swap browser/viewers/terminal — marketplace extensions?); 10/10 UI/UX.
- **Assumptions:**
  - The web cockpit is the primary surface; Obsidian (DirectTransport) and Electron must degrade gracefully, not gate v1.
  - "Folder explorer" refers to `DirectoryPicker` (cwd-selection modal). Finding: it is **not** the right base for a workbench explorer (see §3); the explorer is new UI, but its server/boundary patterns are reused.
  - The generative-UI program has now LANDED on `main` (#127–#136); we build on shipped platform capabilities, not concurrent branches.
  - **Obsidian/DirectTransport byte-serving gap:** `transport.mediaUrl` returns `null` under DirectTransport (no HTTP surface for a webview to fetch). Any viewer/browser needing raw bytes is web-first; Obsidian parity for those needs new in-process plumbing (deferred), consistent with the "web cockpit is primary" framing.
- **Out of scope (v1):**
  - Third-party/marketplace-swappable terminal or browser implementations (see D7).
  - Full IDE ambitions: multi-file IntelliSense (Monaco-class language services), debugging, SCM UI.
  - Arbitrary external-site embedding guarantees (X-Frame-Options makes this impossible in a compliant browser; graceful fallback instead).
  - Mobile terminal beyond degraded/read-mostly.
  - `pip`/floating panel surfaces (deferred with MCP Apps `pip`).

## 3) Codebase Map

- **Right panel** — `apps/client/src/layers/features/right-panel/ui/RightPanelContainer.tsx`, rendered in the app `PanelGroup` (`AppShell.tsx:332-337`); desktop resizable panel, mobile full-width Sheet. Tabs come from the `right-panel` slot in `apps/client/src/layers/shared/model/extension-registry.ts` (`RightPanelContribution { id, title, icon, component, visibleWhen?, priority }`); built-ins registered in `apps/client/src/app/init-extensions.ts:71-92` (`agent-hub` p10, `canvas` p20). Each tab lazy-loads inside `PanelErrorBoundary`. State: `app-store-right-panel.ts` (localStorage-persisted). **The two-tab structure does not need changing — it is already N-tab.**
- **Canvas (post-merge, 7-variant union)** — `UiCanvasContentSchema` (`packages/shared/src/schemas.ts:2126-2191`) is now `url | markdown | json | image | pdf | widget | mcp_app`. `image`/`pdf` carry `src: CanvasMediaSrcSchema` (a path/URL/`data:`, cwd-confined); `widget` carries a `WidgetDocument`; `mcp_app` carries `{serverName, uri}`. `UiStateSchema.canvas.contentType` enum mirrors all 7 (`schemas.ts:2299-2306`). Renderers under `layers/features/canvas/ui/`: `CanvasImageContent.tsx` (`<img>` object-contain, click-to-open-full-size in new tab, **no in-canvas zoom/pan**), `CanvasPdfContent.tsx` (native `<object type=application/pdf>`, **no PDF.js/page controls**), `CanvasWidgetContent.tsx`, plus `CanvasUrlContent.tsx` (the legacy single sandboxed iframe — protocol blocklist, sandbox-token allowlist, default `allow-scripts allow-same-origin allow-popups allow-forms`, **no navigation chrome**). Media classification: `canvas/lib/media-src.ts` (`resolveCanvasMediaSrc`: remote vs `data:` vs local-via-`transport.mediaUrl`; blocks other schemes). **Canvas holds ONE content object at a time** (`use-canvas-persistence.ts`; `open_canvas`/`update_canvas` replace wholesale) — multi-document tab state is net-new (D11).
- **File-read endpoint + Transport (post-merge, #128)** — `GET /api/files/raw` (`routes/files.ts:54-128`): `RawFileQuerySchema {cwd, path}` (`schemas.ts:1481-1488`), double `validateBoundary`, extension allowlist `MEDIA_CONTENT_TYPES` (images + pdf only → 415 otherwise), `createReadStream().pipe(res)` with `nosniff`/`Content-Disposition: inline`/SVG-script-neutering CSP, **no byte-size cap**. Transport method: `mediaUrl(cwd, filePath): string | null` (`transport.ts:466`); `HttpTransport` builds the raw URL (`system-methods.ts:78-81`); **`DirectTransport` returns `null`** (`direct/system-methods.ts:114-116`) — "in-process host has no HTTP surface a webview can fetch." **This is a real Obsidian gap any byte-serving workbench feature inherits.** The route is deliberately narrow (never a general download endpoint) — workbench text/3D/CSV viewing extends its allowlist or adds a sibling route reusing the same boundary pattern.
- **Agent → canvas path** — `control_ui` MCP tool → `ui_command` StreamEvent on session SSE → `stream-manager.ts` (only when `sessionId === attachedSessionId`) → `executeUiCommand` (`ui-action-dispatcher.ts`, skips pushes while `canvasEditing`) → Zustand app-store → render. `UiCommandSchema` (`schemas.ts:2221-2274`) now has **14 actions** (`open_panel`/`close_panel`/`toggle_panel`, sidebar trio, `open_canvas`/`update_canvas`/`close_canvas`, `show_toast`, `set_theme`, `scroll_to_message`, `switch_agent`, `open_command_palette`). Note: `mcp_app` canvas content is set **client-side by `McpAppBlock` on fullscreen**, NOT via an agent `control_ui` payload. **Adding a workbench action (`open_file`/`open_terminal`/`browser_navigate`) is a 3-place recipe:** shared `UiCommandSchema` + shared contract (`ui-tool-contract.ts` `CONTROL_UI_DESCRIPTION`/`CONTROL_UI_INPUT`) + `applyUiCommandToState` (`ui-tools.ts:62-96`, for `get_ui_state` parity); Codex reuses the shared contract via its external `dorkos_ui` stub (`codex-ui-mcp-server.ts`) + event-mapper interception — no separate Codex schema.
- **ui-action return channel (post-merge, #133)** — `POST /api/sessions/:id/ui-action` (`routes/session-ui-action-handler.ts`): validates `UiActionRequestSchema`, formats a runtime-neutral `<ui_action>` block (`ui-widget.ts` `formatUiActionMessage`), injects it as a **fresh turn** via `triggerTurn` (202, streams over `/events`). **This is the established "user did X in the panel → tell the agent" loop — reuse it, don't invent a new one.**
- **MCP-apps host (post-merge, #135) — the sandbox precedent** — shipped: `layers/features/mcp-apps/` (`McpAppFrame`, `McpAppBlock`, `lib/sandbox.ts`, `model/bridge.ts`, consent), server `POST /api/sessions/:id/mcp-app/resource` + `services/mcp-apps/`. Posture (`lib/sandbox.ts`): **`srcDoc` iframe, `sandbox="allow-scripts"` ONLY (never `allow-same-origin` → opaque `"null"` origin)**, injected `<meta>` CSP (`default-src 'none'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'`), `allow` strictly from declared permissions, postMessage bridge origin-checked + attached before load, links via `LinkSafetyModal`. Render-only v1 (app `tools/call` → "not permitted"; no `pip`, no Codex). **The workbench browser + local-HTML serving mirror this posture — but call out the differing threat model: this frame renders already-trusted server-fetched App HTML; a browser navigates arbitrary external URLs and local HTML runs untrusted project files.**
- **Widget tier (post-merge, #131)** — `layers/features/gen-ui/` (`WidgetRenderer`, `WidgetFence` wired into chat `StreamingText.tsx`, node catalog `ui/nodes/`); `dorkos-ui` fences in assistant text. Not directly reused by the workbench, but establishes the fence-content-vs-`control_ui`-control split we must not violate.
- **Folder explorer & markdown editor** — `DirectoryPicker.tsx` is a cwd-selection modal (`GET /api/directory`, directories only, `routes/directory.ts:58`); flat `@`-mention palette in `layers/features/files/` (`GET /api/files`, git-ls-files, 10k cap). **No tree explorer with CRUD exists.** Markdown editor = Blintz (Milkdown WYSIWYG) via `BlintzCanvas.tsx`, canvas-only, save via `useCanvasFileSave` → `transport.writeFile` → `PUT /api/files/content` (409 optimistic-concurrency, atomic write, **never creates files**). Blintz is markdown-only; CodeMirror exists only as its internal dep, not a standalone code editor.
- **File security & worktrees** — path security centralized in `apps/server/src/lib/boundary.ts` (`validateBoundary`: `~` expansion, realpath, null-byte rejection, prefix-collision-safe); reuse for every new file/dir route. Worktrees: `packages/shared/src/workspace.ts`; `workspace-service.ts` `resolveByPath(absPath)` maps cwd → workspace; client `selectedCwd` + `transport.resolveWorkspace()` = the worktree-awareness hook for explorer + terminal.
- **Transport streaming precedent** — `transport.ts` per-domain modules on both impls; existing file methods `listFiles`/`writeFile`/`browseDirectory`/`createDirectory`/`uploadFiles`/`getGitStatus`/`mediaUrl`. `subscribeSession` (`AsyncIterable` + `AbortSignal`) is the shape a PTY byte-stream should follow.
- **Terminal/PTY — confirmed net-new** — no xterm/node-pty anywhere (OpenCode `pty.*` events filtered out unhandled). Fully new: PTY service, byte-stream transport, xterm.js client, resize/signal handling, `open_terminal` action.
- **Extension system** — client slot registry (8 slots incl. `right-panel`); `registerComponent('right-panel', …)` works end-to-end; bundles esbuild-compiled server-side and **loaded unsandboxed** in the app origin (`extension-loader.ts`); `manifest.permissions` is a reserved no-op; marketplace `plugin` packages distribute `.dork/extensions/` bundles. New typed extension-events API (#130, `packages/extension-api/src/extension-events.ts`): curated privacy-safe event kinds, manifest-declared `capabilities.events`, **client-side bridge, no server route** — the pattern for exposing workbench lifecycle hooks (e.g. `file.opened`) to extensions later.

## 4) Research

External research (full report in session; key sources: Cursor 3 Agents Window/Design Mode, Claude Code VS Code extension, Zed Agent Panel/ACP, VS Code Simple Browser + Live Preview, coder/ghostty-web v0.4.0, chrome-devtools-mcp):

- **Emerging agent-cockpit patterns (2025–2026):** chat-primary with agent-driven panels (the agent opens the file/diff/preview, the human doesn't hunt); diff-centric review with per-hunk accept as table stakes; preview panes becoming **bidirectional** (annotate the preview → agent, Cursor Design Mode); worktree/agent isolation surfaced in panel chrome (Zed); dev-server auto-preview (StackBlitz/Codespaces).
- **Terminal:** xterm.js (`@xterm/xterm` + `@xterm/addon-fit` + `@xterm/addon-webgl`) + `node-pty` over WebSocket is the proven stack (it's what VS Code ships). **Ghostty Web is real** (coder/ghostty-web v0.4.0, Dec 2025, MIT, drop-in xterm.js API, WASM ~400KB) **but not launch-ready**: `data:` WASM URL breaks under strict CSP (bit OpenCode: issues #9420/#26001), needs async init, perf unoptimized. Verdict: xterm.js now; ghostty-web is a same-API swap later. ttyd/Wetty are turnkey pages, not embeddable libs.
- **Editor:** CodeMirror 6 over Monaco — ~50–300KB tree-shaken vs ~5MB; materially better mobile/touch/IME (decisive given the mobile mandate); themes are plain JS/CSS (Tailwind-4-friendly); `@codemirror/lang-markdown` + language `Compartment` = one editor, many modes. Monaco only if full TS language services are ever needed (lazy-loaded island, later).
- **Viewers matrix (GitHub/VS Code-informed):** images `<img>` + `react-zoom-pan-pinch`; 3D `<model-viewer>` for glTF/GLB + three.js `STLLoader`/`OBJLoader` for STL/OBJ (GitHub's own approach); PDF native embed or pdf.js; video/audio native elements; CSV papaparse + virtualized table; text/code CM6 read-only (same code path as edit).
- **Local HTML / embedded browser:** the VS Code answer is **local server + iframe** (Live Preview roots an HTTP server at the workspace; Simple Browser iframes it). For DorkOS: an authenticated, boundary-validated static/proxy route on the Express server, iframed same-origin — sidesteps CORS entirely and serves relative assets (answers DOR-98's "do relative assets matter?" with yes, via serving strategy A). For `localhost:<port>` dev servers: reverse-proxy route stripping `X-Frame-Options`/`frame-ancestors`. For arbitrary external sites: **no compliant bypass exists** — iframe best-effort, detect refusal, offer "open in system browser" (+ Electron `<webview>` in desktop later).
- **Not-thinking-of opportunities:** first-class diff viewer; "agent touched this" provenance badges in the explorer; port-detection → auto-offer preview; Chrome DevTools MCP-style bridge (browser pane exposes console/network/screenshot back to the agent — differentiated and on-thesis); notebook rendering; CSP planned up front (ghostty-web + iframes both punish retrofits).
- **Tracker findings:** the remembered ticket is **DOR-98** "Canvas: render local HTML files — serving strategy" (Triage, 2026-06-11) — its option (1), a session-workspace-scoped static endpoint, matches our recommendation. DOR-96/97 (canvas scroll/auto-open) already Done. No existing Linear project fits; a new project should be created with DOR-98 pulled in.

**Potential solution shapes considered:**

1. **Everything-in-canvas** (no new tabs; explorer/browser/terminal all as canvas content types) — Pros: one surface, small schema delta. Cons: canvas is a _stage_ (one document at a time); a terminal or explorer is _persistent tooling_, killed every time the agent pushes a document; overloads `UiCanvasContentSchema`.
2. **New tabs via the existing `right-panel` slot; canvas remains the document stage** — explorer + terminal are persistent sibling tabs; file viewing rides canvas content types (where image/pdf are already in flight); browser = enhanced `url` canvas content with chrome. Pros: zero structural change, matches shipped architecture (#237), tab persistence for free, conflict surface minimized. Cons: content-union coordination with PRs #128/#131.
3. **A single monolithic "Workbench" tab** with its own inner explorer/viewer/terminal/browser layout — Pros: IDE-familiar. Cons: rebuilds tab management inside a tab; fights the slot system; worst mobile story.

**Recommendation: shape 2.** Explorer and Terminal as new `right-panel` contributions; viewers and browser as canvas content types (extending the in-flight `image`/`pdf` work); agent control via `control_ui` extensions.

**Decision round 2 (Dorian, 2026-07-08):**

1. **Diff viewer → fast-follow, confirmed.** Tracked as DOR-212 in the Right-Panel Workbench project (Backlog); sequenced after workbench v1 ships and is verified working.
2. **DevTools bridge → v2, confirmed.** The v1 browser ships as a plain preview; the agent-self-verification bridge (console/network/screenshot exposed to the agent) is deferred to v2 and tracked in Linear so it is not forgotten (see D11). The v1 browser must be architected so the bridge attaches cleanly later (proxy route already sees traffic; console capture is an injected script on served pages) — no v1 rework.
3. **Terminal on Obsidian → hide the tab** via `visibleWhen`, confirmed.
4. **Multi-document canvas → confirmed** (see D11): model canvas state as `openDocuments[] + activeDocumentId` from day one, with a document-tab strip in the canvas header (close buttons, LRU eviction, per-document edit-protection); agent `open_*` commands append-and-activate instead of replace. No split panes / side-by-side in v1 (panel is narrow; poor mobile story). Retrofitting single→multi later would touch every viewer, the persistence layer, and `control_ui` semantics.

**Recommended next step:** SPECIFY (`/flow:specify`) — decompose into 6 execution chunks (file-service foundation → explorer UI → viewer matrix + CM6 editor → browser/local-HTML (DOR-98) → terminal → agent `control_ui` extensions), each as its own tracked issue, worktree, and named agent, with code-review per REVIEW.md before merge.
