## Spec conformance — implementer-delegated decisions

1. **Zod refine mechanics** (spec §1). `SidebarGroupSchema` became a `superRefine`-wrapped schema (base shape kept as an unexported `SidebarGroupShapeSchema`) rather than adding the constraint as a second top-level schema, matching the existing `OperationProgressEventSchema` pattern in `schemas.ts`. Verified empirically that `z.toJSONSchema(UserConfigSchema)` (the conf/Ajv bridge) tolerates the nested refine without throwing — the refine is structurally dropped from the JSON Schema (as expected; arbitrary predicates aren't representable), but validation still runs correctly through `ConfigManager`'s own Zod-level checks.
2. **Migration key composition** (spec §1, task instructions). DOR-339's migration had already claimed the next-unreleased-version key `'0.55.0'`. Rather than adding a duplicate object key (which would silently overwrite DOR-339's migration in a plain object literal — a real footgun), `backfillSmartGroupKindDefaults` was composed into the _same_ `'0.55.0'` entry as a second call after `backfillSidebarSettingsDefaults`, matching the file's own established composite-migration convention (see the `'0.45.0'`/`'0.46.0'`/`'0.48.0'` entries). The `CONFIG_MIGRATIONS` key-invariant regression test (DOR-339) still passes unmodified since no new key was added.
3. **`sortMode: 'manual'` rejection scope** (spec §1). Enforced both at the schema level (the `superRefine` rejects a smart group whose `sortMode` is `'manual'`, including the _default_ value when omitted — callers constructing a smart group must explicitly pass a non-manual `sortMode`) and defensively at the render path (`AgentGroupSection` falls back `manual → recent` for any data that predates the constraint, per the spec's explicit instruction).
4. **`describeRules` wording** (spec §3). The spec's own example ("Codex · active in last hour") is illustrative, not a literal contract. Implemented as "Codex · active in the last hour" (grammatically complete, matches `writing-for-humans`), with English list-joining ("a", "a or b", "a, b, or c") for multi-value fields and three hard-coded phrasings for the UI's three activity-window presets (1h/24h/7d), falling back to a computed "N day(s)"/"N hour(s)" phrase for any other value (defensive — the UI never offers one, but the rule schema doesn't forbid it).
5. **"Multi-select" form controls** (spec §4). No generic multi-select component exists in `shared/ui`. Implemented namespace and status "multi-select" as checkbox groups (consistent with the spec's own "checkbox set per runtime" wording for the runtime field), rather than introducing a new component. Runtime, namespace, and status are all checkbox groups; activity window is a `Select` (single choice + "No limit"); path prefix is a plain `Input`.
6. **Preset chips as popover menu items, not pill chips** (spec §4). `AddAgentMenu` is a flat vertical-button `Popover`, not a chip/tag UI anywhere else in the sidebar. Presets render as additional menu items under a "Smart group" label (consistent with the popover's existing visual language) rather than introducing a new horizontal pill-chip pattern for a single call site.
7. **Drop-rejection hint delivery** (spec §3: "reject with a brief inline hint"). Implemented as a `sonner` toast (`SidebarDnd`'s existing transient-feedback library, already imported by `DashboardSidebar` for error toasts) naming the target group, rather than a bespoke inline banner requiring new lifted state threaded through `AgentGroupSection`. The _classification_ stays pure and fully unit-tested (`reject-smart-group` op in `use-sidebar-dnd.ts`); only the toast call is live-DOM-only and untested by unit tests (acceptable — it is a single `toast.info(...)` call gated on a pure, tested classification).
8. **Non-draggable smart-group member rows** (spec §3: "Member rows inside a smart group are not draggable-out"). `renderRow` gained an optional third parameter `{ draggable?: boolean }`; when `false`, `DashboardSidebar` renders `AgentListItem` directly with the already-exported `DISABLED_SORTABLE_BINDINGS` inert bindings instead of wrapping in `<Sortable>`, and `AgentGroupSection` skips the `SortableList` registration entirely for a smart group's visible rows (there is nothing to reorder — order comes from `sortMode`, never a drag gesture).
9. **`convertSmartGroupToManual` and `agentPaths`** (spec §4, ideation decision 5). Materializes the caller-supplied `currentMembers` (the derived list at click time) verbatim into `agentPaths`, flips `kind` to `'manual'`, and clears `rules` to `undefined` (dropped on the wire by `JSON.stringify`). `name`/`collapsed`/`sortMode`/`muted`/`displayFilter` are left untouched exactly as specified.
10. **`SmartGroupCandidate` data sources** (spec §2). `runtime` reads `agents[path]?.runtime ?? 'claude-code'` (the same default the schema itself uses elsewhere) and `namespace` reads `agents[path]?.namespace ?? null` from the already-fetched `useResolvedAgents` manifest map; `attention` and `lastActivityAt` reuse the sidebar's existing `useAgentAttentionMap` / `useRecentSessions().agentActivity` — no new data fetch.

## Deviations from the frozen spec (all reasoned above or low-risk)

- No standalone "group-create dialog" existed before this spec (DOR-329's manual flow is an inline row, `GroupCreateInput`). Rather than convert the whole create flow into a modal, the manual path is untouched and a new `SmartGroupRuleDialog` (a real `Dialog`, since the rule form needs more surface than an inline row) is used only for smart-group creation ("Custom rules…") and editing. Manual create stays exactly as fast as before; the new chrome is additive.
- `docs/getting-started/configuration.mdx` was not touched, matching the same reasoning DOR-339 documented: `ui.sidebar` is client-only UI preference, not CLI/env-settable config.

## Follow-ups (not blocking)

- The status-flicker watch-item from the spec's Open Questions (hysteresis at the active/idle boundary) is unaddressed — it inherits whatever behavior the DOR-339 attention module has today, per the spec's own framing ("watch-item... add hysteresis there if real use flickers").
- Namespace data population in real fleets is unverified beyond the mesh schema's `namespace` field being optional; the namespace checkbox group simply doesn't render when fewer than 2 distinct namespaces are present, so an unpopulated fleet degrades gracefully (per spec's own watch-item).
- No dev-playground showcase was added for the new smart-group chrome (`SmartGroupRuleDialog`, the rule-form checkboxes) — out of scope for this spec's phases, flagged for a future playground pass per `maintaining-dev-playground`.
- `lastActiveWithinMs` has a small staleness window between store ticks (a candidate's `lastActivityAt`/`attention` is only as fresh as the last render's subscription tick, not a live clock) — PR #384 review confirmed this is accepted v1 behavior per the spec, not a bug; no code change.
