# Canvas File Viewing Overhaul — Specification

- **Id**: 260722-171944 · **Tracker**: DOR-420 · **Created**: 2026-07-22
- **Status**: specified → executing
- **Surfaces**: `apps/client` (canvas feature), `apps/server` (file routes), `packages/shared` (viewer registry, schemas), external `blintz` repo (`/Users/doriancollier/Keep/144/blintz`)

## 1. Problem

Four user-reported defects in the session right-panel Files → Canvas flow, plus one feature request:

1. **Code files fail to open and kill the canvas.** When any canvas viewer throws during render, the error is caught by `PanelErrorBoundary` wrapping the _entire_ `CanvasContent` (`RightPanelContainer.tsx:181`) — tab strip included. The user loses every open tab and cannot navigate.
2. **Markdown files opened from the Files tab cannot be edited.** The pencil toggles `editable` on `<BlintzCanvas>` without a remount key (`CanvasFileContent.tsx`, markdown branch), but blintz captures `editable` at editor construction (`useBlintzEditor.ts:100`, `editableRef` never re-synced). Verified live: `.milkdown [contenteditable]` stays `false` after clicking Edit.
3. **Black-on-black markdown in light theme.** Two blintz-side causes, both verified live (OS dark + app light): (a) blintz's `vars.css` activates dark tokens from the raw `@media (prefers-color-scheme: dark)` query, ignoring the host's explicit light theme (`--crepe-color-background` resolved to `#1b1c1d` while `<html>` had no `.dark` class); (b) the Crepe→Blintz port dropped Crepe's `reset.css` rule that paints `.milkdown` itself (`background: var(--crepe-color-background); color: var(--crepe-color-on-background)`), so the editor never asserts its own colors — dark surface tokens (code blocks, tables) combine with near-black text inherited from the light app.
4. **(Feature)** Expand what the canvas can view: all common software-project file types, plus 3D formats (3MF, PLY, FBX, DAE in addition to today's GLB/glTF/STL/OBJ) and audio/video playback. Unsupported binaries must degrade gracefully in-tab, never crash the canvas.

### Root-cause notes for (1)

Common text formats (`.ts`, `.js`, `.json`, `.yaml`, huge `pnpm-lock.yaml`) open **correctly** on a freshly loaded page against both the dev SPA (:6241) and built cockpit (:4242) — verified via Playwright sweep 2026-07-22. The reported failure is environmental, with two identified trigger classes, both funneling into the boundary-wipes-canvas defect:

- **Stale lazy chunks**: the cockpit tab kept open across a `dev:dogfood` rebuild requests a `CodeMirrorEditor` chunk hash that no longer exists → dynamic import rejects → `React.lazy` throws → whole canvas replaced. (Markdown/images keep working because their chunks were already loaded.)
- **WebGL context failure**: `Model3dViewer`'s `new THREE.WebGLRenderer()` throws synchronously when a context can't be created (GPU off, context exhaustion — xterm/terminal tabs also hold WebGL contexts) → same wipe-out.

The fix is defense-in-depth: contain per-document, retry chunk loads, and guard WebGL — the canvas must survive **any** viewer failure regardless of trigger.

## 4. Acceptance criteria

1. `.ts`/`.js`/`.json`/`.yaml`/dotfiles open in CodeMirror with highlighting; canvas tab strip intact throughout.
2. A viewer that throws (simulated) shows an error card **inside that tab only**, with working Retry; other tabs remain clickable; closing the broken tab works.
3. Files-tab markdown: pencil → editor becomes editable immediately (no remount), typing autosaves via the existing 500 ms debounce + optimistic-concurrency flow; checkmark exits cleanly.
4. OS-dark + app-light renders markdown light (light background, dark text, light code blocks); OS-light + app-dark renders dark. Both verified in-browser.
5. `.stl`/`.obj`/`.glb`/`.3mf`/`.ply`/`.fbx`/`.dae` render in the 3D viewer with orbit controls; WebGL unavailability shows an in-tab message.
6. `.mp3`/`.mp4` play with functional seeking (Range requests observed).
7. A binary (e.g. `.zip`) shows the graceful message; no boundary trip.
8. Unit coverage: canvas boundary behavior, blintz reactive-editable integration (real blintz, not mocked), registry mappings, raw-route MIME + Range, Model3dViewer format dispatch. Blintz repo: jsdom test proving `editable` toggles live.
9. `pnpm verify` green in both repos; changelog fragments in DorkOS for both PRs.
