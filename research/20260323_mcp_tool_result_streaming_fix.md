# MCP Tool Result Streaming — Root Cause Analysis and Fix Approaches

## Research Summary

The root cause of missing MCP tool results during streaming is definitively confirmed: the Claude Code Agent SDK emits `tool_use_summary` messages only for built-in SDK tools (Read, Bash, Edit, etc.), **not** for external MCP tools. For MCP tools, the SDK records results in JSONL via a `user` message containing `tool_result` content blocks, but does NOT emit a corresponding `tool_use_summary` stream event. The `sdk-event-mapper.ts` in DorkOS correctly handles `tool_use_summary` for built-in tools, but there is no mechanism to relay MCP tool results during live streaming. The fix requires intercepting MCP results from the `SDKAssistantMessage` / `SDKUserMessage` pair in the stream, or reading them from JSONL at message completion.

Additionally, the missing `input` field on streamed MCP tool cards is a secondary related issue: `input_json_delta` events DO fire for MCP tools with non-empty inputs, but for the common case of MCP tools with `{}` (empty) inputs, there is no JSON delta to stream — the JSONL records `'{}'` but this is generated from `block.input` during transcript parsing, not from streaming.

## Key Findings

### 1. The `tool_use_summary` Gap Between Built-In and MCP Tools

The `SDKToolUseSummaryMessage` type is confirmed in the SDK:

```typescript
type SDKToolUseSummaryMessage = {
  type: 'tool_use_summary';
  summary: string;
  preceding_tool_use_ids: string[];
  uuid: UUID;
  session_id: string;
};
```

This event is the mechanism by which the SDK delivers tool result summaries to the streaming consumer. The SDK's internal `cli.js` source confirms that `tool_use_summary` messages are generated by a _summarization pass_ that processes tool results from the Anthropic API and emits a brief natural-language summary.

**Critical distinction:**

- **Built-in tools** (Read, Bash, Glob, etc.): The Claude Code CLI intercepts the `tool_result` blocks returned by the Anthropic API and, after processing, emits a `tool_use_summary` event with `{ summary: "<brief summary>", preceding_tool_use_ids: [...] }`.
- **External MCP tools**: The Claude Code CLI executes the MCP call, receives the result, and writes it to JSONL (as a `user` message with `tool_result` blocks), but does NOT emit a corresponding `tool_use_summary` stream event. The MCP result is stored in JSONL only.

This behavior is consistent with the observation that built-in tool results appear in streamed `ToolCallCard` components while MCP tool results do not.

**Why the summary approach won't cover MCP**: The `tool_use_summary` mechanism was designed for the built-in tools that the CLI orchestrates directly. External MCP tool calls go through a different execution path in the CLI — the result is forwarded to the Anthropic API as a `tool_result` in the user turn, then the API response continues. The CLI does not generate a summary for this path.

### 2. The `SDKAssistantMessage` Path Carries the Full MCP Result

The key insight is that when `includePartialMessages: true` is set (which DorkOS does use), the SDK emits both:

- `stream_event` messages (the `SDKPartialAssistantMessage` type) during streaming
- `assistant` type messages (the `SDKAssistantMessage` type) with the **completed** `BetaMessage` after each turn

The `SDKAssistantMessage` contains the full `BetaMessage` from Anthropic, which includes:

- The assistant turn content blocks (text, thinking, tool_use)
- For each `tool_use` block: the full `id`, `name`, and `input` fields

Additionally, a `user` message (`SDKUserMessage`) follows the assistant turn containing the `tool_result` blocks from the MCP server response.

**Current sdk-event-mapper.ts behavior**: The mapper handles `stream_event` messages (partial streaming) but falls through to the catch-all debug log for `assistant` and `user` type messages. These messages are never processed for tool results.

The complete turn sequence in the SDK stream for an MCP tool call is:

```
1. system/init
2. stream_event: content_block_start (tool_use) → tool_call_start
3. stream_event: content_block_delta (input_json_delta) → tool_call_delta [may be absent for empty inputs]
4. stream_event: content_block_stop → tool_call_end [NO result data]
5. assistant message (SDKAssistantMessage) with BetaMessage containing tool_use block [NOT CURRENTLY PROCESSED]
6. user message (SDKUserMessage) with tool_result blocks containing MCP result [NOT CURRENTLY PROCESSED]
7. stream_event: content_block_delta (text_delta) [follow-up text]
8. tool_use_summary → ABSENT for MCP tools
9. result/success → done
```

### 3. Input Missing for Empty-Input MCP Tools

The JSONL shows `'{}'` for MCP tool inputs with empty parameters, while streamed cards show empty input. The cause:

- During streaming: `input_json_delta` events only fire when there IS partial JSON to stream. For MCP tools with empty `{}` input, the Anthropic API may emit a single empty delta or no delta at all.
- During transcript loading: `transcript-parser.ts` reads `block.input` (the fully assembled object) and calls `JSON.stringify(block.input)` → `'{}'`. This reconstructed string is never sent during streaming.

This is a separate rendering issue from the missing result. The input display can be fixed independently by checking for empty deltas.

### 4. The `SDKStreamlinedToolUseSummaryMessage` Internal Type

The SDK's internal `StdoutMessage` union references `coreTypes.SDKStreamlinedToolUseSummaryMessage` — this is an **internal** message type used between the CLI process and the SDK wrapper, never exposed to SDK consumers. This confirms there is no alternative pathway for MCP results to appear as `tool_use_summary` events.

## Detailed Analysis

### Approach 1: Read MCP Results from the `assistant`/`user` SDK Message Pair

**How it works:** The `sdk-event-mapper.ts` currently ignores `SDKAssistantMessage` (type: `'assistant'`) and `SDKUserMessage` (type: `'user'`) messages. These ARE emitted by the SDK for every turn, including MCP turns. The `user` message that follows an MCP tool call contains `tool_result` content blocks with the full result text.

```typescript
// In mapSdkMessage — add handling for user messages with tool_result blocks
if (message.type === 'user') {
  const userMsg = message as SDKUserMessage;
  const content = userMsg.message?.content;
  if (Array.isArray(content)) {
    for (const block of content) {
      if (block.type === 'tool_result' && block.tool_use_id) {
        const resultText = extractToolResultText(block.content);
        yield {
          type: 'tool_result',
          data: {
            toolCallId: block.tool_use_id,
            toolName: toolState.toolNameById.get(block.tool_use_id) ?? '',
            result: resultText,
            status: 'complete',
          },
        };
      }
    }
  }
  return;
}
```

**Key consideration**: The `SDKUserMessage` contains `MessageParam` from `@anthropic-ai/sdk`, where the content can be `string | ContentBlockParam[]`. The `ContentBlockParam` union includes `ToolResultBlockParam` with `tool_use_id` and `content`. This is the exact same format used in `transcript-parser.ts` (via the JSONL `user` record format).

**When does this fire?**: For both built-in tools AND MCP tools, the SDK emits a `user` message after the assistant turn completes. However, for built-in tools, `tool_use_summary` fires BEFORE the next turn begins. For MCP tools, the `user` message IS the delivery mechanism.

**Risk**: Built-in tools currently have their results handled by `tool_use_summary`. If we also handle `user` messages, we'd emit `tool_result` TWICE for built-in tools. This requires deduplication logic: skip `user` message tool results if a `tool_use_summary` has already fired for those IDs.

**Deduplication strategy**: Track which `toolCallId`s have already been resolved via `tool_use_summary` in `toolState.toolNameById` (or add a `toolState.resolvedToolIds: Set<string>`). When a `user` message arrives, only emit `tool_result` for IDs not already resolved.

**Pros:**

- Correct and complete — handles both input AND result from a single source
- Works immediately with existing SDK version
- No dependency on JSONL filesystem reads during streaming
- Real-time delivery (fires when the SDK delivers the `user` turn back)
- The `assistant` message also carries complete `input` objects, so can fix the empty-input issue too

**Cons:**

- Requires understanding of `MessageParam` / `ContentBlockParam` types from `@anthropic-ai/sdk`
- Need to handle the deduplication for built-in tools carefully
- The `user` message may arrive slightly after the `tool_call_end` event (same turn, but ordering is deterministic)
- Requires updating `toolState` to track resolved IDs

**Complexity:** Low-Medium
**Reliability:** High — this is the data the SDK actually delivers
**Latency:** Zero — fires synchronously within the streaming loop

### Approach 2: Fallback JSONL Read on Session Completion

**How it works:** When the `result` message arrives (signaling session completion), read the JSONL transcript file for the current session, find the most recently written `user` message containing `tool_result` blocks, and emit retroactive `tool_result` events for any MCP tools that didn't receive results.

```typescript
// In the result handler, after emitting 'done':
const transcript = await readTranscript(session.sdkSessionId);
const unresolvedIds = getUnresolvedToolIds(toolState);
for (const [toolCallId, result] of getResultsFromTranscript(transcript, unresolvedIds)) {
  yield { type: 'tool_result', data: { toolCallId, result, status: 'complete', toolName: '' } };
}
```

**Pros:**

- Simple — reads from a source that's confirmed to have the data
- No deduplication risk (only fires for unresolved IDs)

**Cons:**

- Delayed — fires after the `done` event, breaking the expectation that results arrive before completion
- Requires filesystem access in the streaming path
- Adds I/O latency to every session completion
- The JSONL may not be fully flushed when `result` fires (race condition)
- DorkOS already has `transcript-reader.ts` but it's not in the hot streaming path

**Complexity:** Medium
**Reliability:** Medium (race condition risk)
**Latency:** High — results appear after the session ends

### Approach 3: Extract Results from the `SDKAssistantMessage` (Input Fix Only)

**How it works:** The `SDKAssistantMessage` contains `message.content` (a `BetaMessage`) which is the completed assistant turn. This includes the full `tool_use` blocks with complete `input` objects (already assembled from streaming deltas). Processing this message can retroactively fix the missing input on MCP tool call parts.

```typescript
if (message.type === 'assistant') {
  const assistantMsg = message as SDKAssistantMessage;
  const content = assistantMsg.message?.content;
  if (Array.isArray(content)) {
    for (const block of content) {
      if (block.type === 'tool_use') {
        // Retroactively set the completed input on an already-started tool call
        yield {
          type: 'tool_call_input_complete',  // new event type needed
          data: { toolCallId: block.id, input: JSON.stringify(block.input) },
        };
      }
    }
  }
  return;
}
```

This only fixes the input display, not the result. It's a partial fix that would need to be combined with Approach 1 or 2 for the result.

**Pros:**

- Fixes the missing input issue without new event types if handled as a delta backfill
- Clean separation of concerns

**Cons:**

- Incomplete — doesn't fix the missing result
- Would require adding a new SSE event type or overloading an existing one

**Complexity:** Low
**Reliability:** High

### Approach 4: Use PostToolUse Hook to Capture MCP Results

**How it works:** Configure a `PostToolUse` hook in the SDK options that intercepts every tool execution after completion. For MCP tools, the `tool_response` field in `PostToolUseHookInput` contains the raw result.

```typescript
hooks: {
  PostToolUse: [{
    matcher: 'mcp__.*',  // regex matcher for MCP tool names
    hooks: [async (input) => {
      // input.tool_response is the raw MCP tool result
      // Inject it back into the stream somehow...
      return {};
    }],
  }],
}
```

**Problem**: Hooks are async callbacks that return `HookJSONOutput`. They run in the SDK's hook pipeline and cannot directly yield SSE events to the streaming consumer. There's no mechanism to pass data from a hook callback back into the `query()` AsyncGenerator stream.

**Possible workaround**: The hook callback signature does support `systemMessage` in its return, which gets injected into Claude's context. But this is model context injection, not SSE event emission. A side-channel approach (e.g., putting results into a shared store that the stream consumer polls) would work but introduces significant complexity.

**Cons:**

- No clean way to bridge from hook callback to SSE event
- Side-channel approach is fragile and introduces ordering issues
- Significantly increases code complexity

**Complexity:** High
**Reliability:** Low (side-channel pattern)

## Recommended Fix: Approach 1 (Process `SDKUserMessage` for MCP Tool Results)

### Implementation Plan

**Step 1: Update `ToolState` to track resolved tool result IDs**

In `agent-types.ts`, add a `resolvedResultIds: Set<string>` field to `ToolState` to track which tool calls have already had their results delivered (via `tool_use_summary`).

**Step 2: Update `tool_use_summary` handling to mark resolved IDs**

In the existing `tool_use_summary` handler in `sdk-event-mapper.ts`:

```typescript
if (message.type === 'tool_use_summary') {
  const summary = message as { summary: string; preceding_tool_use_ids: string[] };
  for (const toolUseId of summary.preceding_tool_use_ids) {
    toolState.resolvedResultIds.add(toolUseId);  // NEW: mark as resolved
    yield {
      type: 'tool_result',
      data: {
        toolCallId: toolUseId,
        toolName: toolState.toolNameById.get(toolUseId) ?? '',
        result: summary.summary,
        status: 'complete',
      },
    };
  }
  return;
}
```

**Step 3: Add `SDKUserMessage` processing for tool results**

After the `tool_use_summary` handler, add:

```typescript
// Handle user messages containing tool_result blocks (MCP tool results)
// Built-in tool results arrive via tool_use_summary; MCP results arrive here.
if (message.type === 'user') {
  const userMsg = message as { message?: { content?: unknown } };
  const content = userMsg.message?.content;
  if (Array.isArray(content)) {
    for (const block of content as Array<Record<string, unknown>>) {
      if (block.type === 'tool_result' && typeof block.tool_use_id === 'string') {
        // Skip if this tool was already resolved by tool_use_summary (built-in tools)
        if (toolState.resolvedResultIds.has(block.tool_use_id)) continue;
        const resultContent = block.content;
        const resultText =
          typeof resultContent === 'string'
            ? resultContent
            : Array.isArray(resultContent)
              ? (resultContent as Array<Record<string, unknown>>)
                  .filter((b) => b.type === 'text')
                  .map((b) => b.text as string)
                  .join('\n')
              : '';
        if (resultText) {
          yield {
            type: 'tool_result',
            data: {
              toolCallId: block.tool_use_id,
              toolName: toolState.toolNameById.get(block.tool_use_id) ?? '',
              result: resultText,
              status: 'complete',
            },
          };
        }
      }
    }
  }
  return;
}
```

**Step 4: Fix missing input for zero-parameter MCP tools**

Also process the `assistant` message to backfill input for tool_use blocks with empty input:

```typescript
// Handle assistant messages to backfill completed tool inputs (especially empty-input MCP tools)
if (message.type === 'assistant') {
  const assistantMsg = message as { message?: { content?: unknown } };
  const content = assistantMsg.message?.content;
  if (Array.isArray(content)) {
    for (const block of content as Array<Record<string, unknown>>) {
      if (block.type === 'tool_use' && typeof block.id === 'string') {
        const inputStr = JSON.stringify(block.input ?? {});
        // Only emit if we have meaningful input that wasn't already streamed
        if (inputStr !== '{}' || !toolState.toolNameById.get(block.id)) continue;
        yield {
          type: 'tool_call_input_complete',
          data: {
            toolCallId: block.id,
            input: inputStr,
          },
        };
      }
    }
  }
  return;
}
```

Wait — `tool_call_input_complete` is a new event type not in the shared schema. A simpler approach: don't add a new event type. Instead, backfill by emitting a `tool_call_delta` with the full JSON when `input_json_delta` events produced an empty string. The client-side `handleToolCallDelta` already appends to existing input, so sending `'{}'` as a delta after the stream would set the input correctly.

Actually the cleanest approach for the empty input: emit a single `tool_call_delta` from the `user` message processing when we detect the tool had no streamed input. This avoids new event types entirely.

**Step 5: Client-side — no changes needed**

`handleToolResult` on the client already handles the `tool_result` event correctly. Once the server correctly emits `tool_result` for MCP tools, the client will display results without any changes.

## Security and Performance Considerations

### MCP Result Size Limits

MCP tool results can be arbitrarily large (e.g., `mcp__dorkos__mesh_list` returning 11KB of JSON per the bug report). The current `tool_use_summary` path delivers a brief summary (AI-generated), not raw result. The `user` message path delivers the **full raw result**.

Implications:

- The existing `tool_result` SSE event has no size limit enforcement
- Large results (>100KB) will inflate the SSE stream significantly
- The client's `existing.result = tc.result` in `handleToolResult` stores the full string in React state, which affects memory

**Recommendation**: Implement a `MAX_TOOL_RESULT_LENGTH` constant (e.g., 50KB) and truncate with a `[truncated: X chars]` suffix if exceeded. This mirrors what the transcript-parser already does with `extractToolResultContent`.

### Streaming vs Batching

MCP results arrive as a complete `user` message (not as streaming deltas). This means the entire result must be available before the `tool_result` event fires. For MCP tools that return very large payloads, this is a step backward from built-in tools where `tool_use_summary` fires quickly with a compressed summary.

However, since DorkOS already accepts this latency for history-loaded sessions (JSONL parsing), it's an acceptable tradeoff for streaming sessions. Users still see the result as soon as the MCP call completes.

### No Rate-Limiting Concerns

The `user` message events come from the SDK's own iteration loop, not from additional API calls. Processing them adds negligible CPU overhead to the existing streaming pipeline.

## Research Gaps and Limitations

1. **SDK version drift**: The installed SDK is version `0.2.58`. The `SDKStreamlinedToolUseSummaryMessage` type referenced in the internal `StdoutMessage` union is not exposed to SDK consumers and its behavior may change in future versions. If Anthropic adds `tool_use_summary` for MCP tools in a future release, the `resolvedResultIds` deduplication handles this correctly.

2. **Empty input confirmation**: The hypothesis that `input_json_delta` doesn't fire for empty `{}` inputs is based on code analysis, not a live trace. The actual behavior should be verified by adding temporary debug logging to the event mapper.

3. **Replay messages**: The SDK also emits `SDKUserMessageReplay` (type: `user` with `isReplay: true`) when resuming sessions. These should NOT be processed as new tool results — they're replayed history. The handler should check for `!message.isReplay` before processing.

## Contradictions and Disputes

- The investigation brief suggests step 4 fires for built-in tools. This is confirmed by the existing `sdkTodoWrite` scenario in `sdk-scenarios.ts` which includes a `tool_use_summary` yield after the tool call, and the DorkOS code at `sdk-event-mapper.ts:294-308`. The dispute is whether this fires for MCP tools — all evidence indicates it does NOT.

- The comment in `stream-tool-handlers.ts` at line 96 reads: `// Set completedAt if not already set (MCP tools complete here, not via tool_result)` — this confirms the codebase already _knows_ MCP tools don't receive `tool_result` events. This is intentional behavior that was accepted rather than fixed.

| Issue                                | Root Cause                                                                                          | Fix Location                                                            | Effort  |
| ------------------------------------ | --------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- | ------- |
| MCP tool result missing              | SDK emits `tool_result` in `SDKUserMessage`, not `tool_use_summary`; mapper ignores `user` messages | `sdk-event-mapper.ts`: add `user` message handler                       | Low     |
