# A2A Protocol v0.3 → v1.0 Breaking Changes and SDK Support Status

## Research Summary

The A2A Protocol specification at `a2a-protocol.org` is currently at **v1.0.0** (the first
production-stable release). The official JavaScript SDK `@a2a-js/sdk` is at **v0.3.13**,
published 2026-03-16, and implements the **v0.3.0 spec only**. There is no v1.0-compliant JS
SDK release yet — the CHANGELOG contains no v1.0 entries. v1.0 introduces nine categories of
breaking changes from v0.3.0, with the most impactful being enum value renaming
(SCREAMING_SNAKE_CASE), Part type unification, method renaming, AgentCard restructuring, and
the removal of the `final` and `kind` fields from stream events. A progressive migration path
is available via dual-protocol AgentCard advertisement.

---

## Key Findings

1. **Protocol version at a2a-protocol.org**: `1.0.0` — clearly labeled as "Latest Released
   Version" on the specification page, with a permalink at `a2a-protocol.org/v1.0.0/specification`.

2. **SDK version on npm**: `@a2a-js/sdk` latest is `0.3.13` (published 2026-03-16). The
   CHANGELOG lists only v0.2.x and v0.3.x entries. No v1.0 protocol support has been shipped.

3. **SDK implements v0.3.0 spec**: The `protocolVersion` field in Agent Cards generated by the
   SDK hardcodes `"0.3.0"`. All type names, enum values, and method names match the v0.3.0 spec,
   not v1.0.

4. **No v1.0 SDK migration in progress (visible externally)**: As of 2026-03-22, the GitHub
   CHANGELOG has no pre-release or alpha entry for v1.0 protocol support. The gap between spec
   and SDK is deliberate — the spec team and SDK team are separate contributors.

5. **Progressive migration is possible**: v1.0 AgentCard allows advertising both `"0.3.0"` and
   `"1.0.0"` under `supportedInterfaces[]`, so clients can migrate incrementally without a
   hard cutover.

## Detailed Breaking Changes: v0.3.0 → v1.0.0

### 1. Method / Operation Renaming

All JSON-RPC method names changed to PascalCase RPC-style names:

| v0.3.0 method                        | v1.0.0 method                      |
| ------------------------------------ | ---------------------------------- |
| `message/send`                       | `SendMessage`                      |
| `message/stream`                     | `SendStreamingMessage`             |
| `tasks/get`                          | `GetTask`                          |
| `tasks/cancel`                       | `CancelTask`                       |
| `tasks/resubscribe`                  | `SubscribeToTask`                  |
| `agent/getAuthenticatedExtendedCard` | `GetExtendedAgentCard`             |
| `tasks/pushNotificationConfig/set`   | `CreateTaskPushNotificationConfig` |
| `tasks/pushNotificationConfig/get`   | `GetTaskPushNotificationConfig`    |
| (no equivalent)                      | `ListTaskPushNotificationConfigs`  |
| (no equivalent)                      | `DeleteTaskPushNotificationConfig` |

### 2. Enum Values: kebab-case → SCREAMING_SNAKE_CASE

All enum values changed format for ProtoJSON compliance:

**TaskState:**

| v0.3.0             | v1.0.0                        |
| ------------------ | ----------------------------- |
| `"submitted"`      | `"TASK_STATE_SUBMITTED"`      |
| `"working"`        | `"TASK_STATE_WORKING"`        |
| `"input-required"` | `"TASK_STATE_INPUT_REQUIRED"` |
| `"auth-required"`  | `"TASK_STATE_AUTH_REQUIRED"`  |
| `"completed"`      | `"TASK_STATE_COMPLETED"`      |
| `"failed"`         | `"TASK_STATE_FAILED"`         |
| `"canceled"`       | `"TASK_STATE_CANCELED"`       |
| `"rejected"`       | `"TASK_STATE_REJECTED"`       |

**MessageRole:**

| v0.3.0    | v1.0.0         |
| --------- | -------------- |
| `"user"`  | `"ROLE_USER"`  |
| `"agent"` | `"ROLE_AGENT"` |

### 3. Part Type Unification (Most Impactful Structural Change)

v0.3.0 uses a discriminated union with a `kind` field:

```typescript
// v0.3.0 — kind discriminator
type Part = TextPart | FilePart | DataPart;

interface TextPart {
  kind: 'text';
  text: string;
  metadata?: Record<string, unknown>;
}
interface FilePart {
  kind: 'file';
  file: FileWithUri | FileWithBytes;
  metadata?: Record<string, unknown>;
}
interface DataPart {
  kind: 'data';
  data: Record<string, unknown>;
  metadata?: Record<string, unknown>;
}
```

v1.0 removes `TextPart`, `FilePart`, and `DataPart` as named types entirely. A single `Part`
type uses member-presence discrimination:

```typescript
// v1.0 — member presence discrimination (no kind field)
interface Part {
  text?: string; // presence of this field → text part
  file?: FileContent; // presence of this field → file part
  data?: Record<string, unknown>; // presence → data part
  metadata?: Record<string, unknown>;
}
```

Code checking `part.kind === 'text'` must be rewritten to `'text' in part` or `part.text !== undefined`.

### 4. Stream Event Discrimination

v0.3.0 uses a `kind` field on stream events:

```typescript
// v0.3.0
{ kind: 'status-update', taskId, contextId, status, final: boolean }
{ kind: 'artifact-update', taskId, contextId, artifact, append?, lastChunk? }
```

v1.0 uses wrapper objects, with no `kind` field and no `final` field:

```typescript
// v1.0
{
  taskStatusUpdate: {
    (taskId, contextId, status);
  }
} // wrapper-based
{
  taskArtifactUpdate: {
    (taskId, contextId, artifact);
  }
} // wrapper-based
```

The `final: boolean` field is removed. Stream closure is signaled by the SSE transport
mechanism directly (connection close), not by a field value.

### 5. AgentCard Structural Reorganization

Several top-level AgentCard fields were removed, consolidated, or moved:

| v0.3.0 field                        | v1.0 change                                               |
| ----------------------------------- | --------------------------------------------------------- |
| `url` (top-level string)            | Removed from root; moved into `supportedInterfaces[].url` |
| `protocolVersion` (top-level)       | Moved to `supportedInterfaces[].protocolVersion`          |
| `preferredTransport`                | Removed; replaced by `supportedInterfaces[]` ordering     |
| `additionalInterfaces[]`            | Removed; merged into `supportedInterfaces[]`              |
| `supportsAuthenticatedExtendedCard` | Moved to `capabilities.extendedAgentCard`                 |

New `supportedInterfaces[]` structure in v1.0:

```typescript
interface AgentInterface {
  url: string;
  protocolBinding: 'JSONRPC' | 'REST' | 'GRPC';
  protocolVersion: string; // "0.3.0" or "1.0.0"
}
```

This `protocolVersion` per-interface field is how dual-protocol advertisement works.

### 6. Error Handling Standard

| v0.3.0                   | v1.0                                            |
| ------------------------ | ----------------------------------------------- |
| RFC 9457 Problem Details | `google.rpc.Status` with `google.rpc.ErrorInfo` |

Errors must now include a `details` array with at least one `google.rpc.ErrorInfo` entry.

### 7. Pagination Model

| v0.3.0                | v1.0                                           |
| --------------------- | ---------------------------------------------- |
| Page-based pagination | Cursor-based pagination via `nextCursor` field |

### 8. HTTP URL Path Simplification

| v0.3.0                       | v1.0                      |
| ---------------------------- | ------------------------- |
| `POST /v1/message:send`      | `POST /message:send`      |
| `POST /v1/tasks/{id}:cancel` | `POST /tasks/{id}:cancel` |

The `/v1` prefix is removed from all HTTP+JSON binding paths.

### 9. Push Notification Config ID Format

| v0.3.0                                                             | v1.0                                     |
| ------------------------------------------------------------------ | ---------------------------------------- |
| Compound path: `tasks/{taskId}/pushNotificationConfigs/{configId}` | Separate fields: `task_id` + `config_id` |

### 10. OAuth Flow Removals

The implicit and password OAuth flows were removed from the supported security schemes.
Only authorization_code, clientCredentials, and deviceCode remain.

## SDK v1.0 Support Status

### Current State (as of 2026-03-22)

| Dimension                      | Status                                  |
| ------------------------------ | --------------------------------------- |
| Latest npm version             | `0.3.13` (published 2026-03-16)         |
| Protocol version implemented   | `v0.3.0`                                |
| v1.0 support in CHANGELOG      | None — no alpha/beta/RC entries         |
| v1.0 support planned           | Not publicly documented with a timeline |
| GitHub CHANGELOG max version   | `0.3.13` — no 0.4.x or 1.x entries      |
| `protocolVersion` in AgentCard | Hardcoded `"0.3.0"` by SDK              |

### What This Means for DorkOS

Any DorkOS code using `@a2a-js/sdk` today:

- Is running v0.3.0 semantics (lowercase enum strings, `kind` discriminators, `final` field)
- Will break if connecting to a v1.0-only server
- Is safe when connecting to a v0.3.0 server or a dual-protocol server that advertises `"0.3.0"` support in `supportedInterfaces[]`
- Should import `TaskState` as an enum rather than raw strings to make the eventual migration to `TASK_STATE_*` values a single-line change

### Migration Path When SDK Upgrades

The v1.0 spec provides a progressive migration strategy. Once an SDK version is released
that supports v1.0:

1. **AgentCard**: Add a v1.0 entry to `supportedInterfaces[]` alongside the existing v0.3.0 entry
2. **Enum values**: Change all `TaskState.Submitted` → `TaskState.Submitted` (the enum import
   stays the same; the SDK will update the string value underneath)
3. **Part type checks**: Rewrite `p.kind === 'text'` → `'text' in p`
4. **Stream events**: Rewrite `event.kind === 'status-update'` → `'taskStatusUpdate' in event`
5. **Remove `final` field usage**: Stop checking `event.final`; detect stream end via SSE close
6. **AgentCard fields**: Remove `url`, `protocolVersion`, `preferredTransport` from root and
   restructure into `supportedInterfaces[]`
7. **Error handling**: Switch from RFC 9457 to `google.rpc.Status` error format

## Migration Guidance Available

The official migration guide is at [a2a-protocol.org/latest/whats-new-v1/](https://a2a-protocol.org/latest/whats-new-v1/).

The A2A v1.0 announcement explicitly states: "AgentCard has evolved in a backward-compatible
way and now allows agents to advertise support for both existing v0.3 protocol behavior and
v1.0 simultaneously, enabling clients to migrate progressively rather than through a single
cutover."

The Quarkus blog (covering the Java SDK 1.0.0.Alpha1) provides the most concrete v1.0
migration code examples available as of this date.

## Research Gaps

1. **SDK v1.0 release timeline**: No public roadmap or GitHub issue tracking when `@a2a-js/sdk`
   will implement v1.0. The Python SDK (`a2a-python`) may move first — worth watching its
   CHANGELOG for signal.

2. **gRPC binding changes**: This report focused on JSON-RPC and HTTP bindings. The gRPC
   `.proto` schema changes between v0.3.0 and v1.0 were not researched.

3. **`@a2a-js/sdk` internal development branch**: There may be a `feat/v1.0` branch on GitHub
   not reflected in the CHANGELOG yet. GitHub was inaccessible during this research.
