# Animated Mood Faces — Lottie Emoji Sets vs. Bespoke Motion SVG for the gen-UI `mood` Widget

## Research Summary

The gen-UI `mood` widget (`apps/client/src/layers/features/gen-ui/ui/nodes/MoodNode.tsx`) draws 8 emotions as hand-built 64×64 SVG faces animated with `motion/react`. The founder's instinct — "maybe there's a Lottie set or React emotion library" — points at **Google's Noto Animated Emoji** (CC BY 4.0, all 8 emotions map to real glyphs, ~28–30KB raw JSON each) as the obvious pre-built candidate, rendered via **`@lottiefiles/dotlottie-react`** (self-hostable WASM, no CDN dependency) rather than `lottie-react` (heavier, pure-JS, no `.lottie` compression).

The catch is aesthetic, not technical: Noto (and Fluent) emoji are full-saturation, glossy, fixed-palette glyphs — they cannot inherit `currentColor`, cannot tint for dark mode or brand accent, and visually clash with DorkOS's flat, monochrome, `--muted-foreground`-driven "calm tech" icon language (everything else in the product, including the `@dorkos/icons` registry, is single-color line/glyph work). Dropping a glossy yellow cartoon face into that system reads as a foreign object, not a showpiece — the opposite of what a delight-pack widget should do.

**Recommendation: keep and elevate the bespoke motion-SVG faces** (Option 4 below) using squash-and-stretch/kawaii-CSS motion principles on top of the widget-motion vocabulary the codebase already has. This is genuinely a one-day job, costs ~0 bytes of new asset weight, has zero licensing exposure, and is the only option that respects dark mode / brand theming and the existing `useWidgetMotion` reduced-motion gate for free. Noto Animated Emoji via `dotlottie-react` is documented here as a credible **runner-up** for a future "make DorkOS feel more universally friendly" push, but it is a multi-day lift (license notice, asset pipeline, self-hosted WASM wiring, and very likely a custom flat/mono treatment to make the emoji not clash) — out of scope for a one-day pass.

## Detailed Analysis

### 1. Google Noto Animated Emoji

**Source**: `googlefonts/noto-emoji-animation` (catalog site: `googlefonts.github.io/noto-emoji-animation`), individual files served at `https://fonts.gstatic.com/s/e/notoemoji/latest/{codepoint}/lottie.json`.

**License — verify this exactly before adopting**: the _font_ files in `googlefonts/noto-emoji` are SIL Open Font License 1.1. The **animated emoji artwork specifically** (the Lottie/GIF/WebP exports) is licensed separately under **CC BY 4.0**, per Google's own developer blog announcement and confirmed by an open (as of this writing, apparently still without a definitive maintainer reply) attribution-clarification thread, [google/fonts#7011](https://github.com/google/fonts/issues/7011). CC BY 4.0 legally requires attribution — practically this means a "Animated emoji artwork © Google, licensed under CC BY 4.0" line in a third-party-licenses/credits surface (DorkOS doesn't currently have one — would need to be created; see `research gap` below). This is a real but small compliance task, not a blocker, but it is friction the bespoke-SVG option doesn't have at all.

**Glyph coverage** — all 8 target emotions map cleanly to real, high-quality Noto animated glyphs (verified present in the catalog's `data/api.json`):

| DorkOS emotion | Candidate Noto glyph                             | Codepoint         |
| -------------- | ------------------------------------------------ | ----------------- |
| happy          | 😊 smiling face with smiling eyes                | `1f60a`           |
| thinking       | 🤔 thinking face                                 | `1f914`           |
| celebrating    | 🥳 partying face (or 🎉 party popper)            | `1f973` / `1f389` |
| sheepish       | 😅 grinning face with sweat (or 😳 flushed face) | `1f605` / `1f633` |
| determined     | 😤 face with steam from nose                     | `1f624`           |
| surprised      | 😲 astonished face (or 😮 face with open mouth)  | `1f632` / `1f62e` |
| sad            | 😢 crying face                                   | `1f622`           |
| love           | 😍 smiling face with heart-eyes                  | `1f60d`           |

**Per-file size**: fetched `1f60a` (smiling face with smiling eyes) directly — a **1024×1024 composition, 159 frames @ 60fps (~2.65s loop), 13 layers** (face base, two animated eye groups with blink/squint, multi-state morphing mouth path, blush opacity fades, null-layer bone rig), raw JSON size **~28–30KB**. Extrapolating, 8 files land around **~230KB raw**; Lottie JSON is highly repetitive keyframe data and compresses well under gzip/brotli, but exact post-compression size should be measured per file before committing to a byte budget — don't take the raw number as the shipped cost.

**Offline/self-hosting**: fully offline-viable — the JSON files are static assets, checked into the repo like any other bundled asset; no runtime fetch needed once vendored. (The renderer's own network dependency is a separate concern, covered under `dotlottie-react` below.)

**Theming**: this is the disqualifying property for this specific widget. Noto animated emoji are baked full-color raster-style vector art (yellow face, gradient blush, glossy highlights) — there's no `currentColor` hook, no way to make it respect `--muted-foreground` or a brand accent, and no clean way to re-skin it for dark mode beyond "it still renders, unchanged." The existing widget's whole visual strategy is a single-color face on a `bg-muted` circle that inherits `text-foreground`/`text-status-error` — Noto emoji breaks that contract completely.

**Honest aesthetic judgment**: in isolation, the Noto set is excellent — genuinely some of the best free animated emoji work available, smooth, characterful, expressive. Inside DorkOS's flat, neutral, "operator control panel" surface (per `contributing/design-system.md` and the `@dorkos/icons` monochrome registry), a full-color glossy 3D-ish emoji face next to plain-text chat content and single-color status icons will read as a jarring, off-brand sticker — closer to a consumer chat app than a professional cockpit. It would need a custom flat mono re-skin to fit, which erases most of the reason to use a pre-built set in the first place.

### 2. Lottie runtimes for React: `lottie-react` vs. `@lottiefiles/dotlottie-react`

**`lottie-react`** (v2.4.1) is a thin React wrapper around `lottie-web`. Unpacked package size **233,648 bytes** (~228KB); its single dependency `lottie-web` (`^5.10.2`) is the actual cost driver — historically measured at **~237.5KB minified / ~60.5KB minified+gzip**. It's pure JS/SVG-DOM rendering (no WASM), broad browser support, easy to lazy-`import()` per-widget so sessions that never render a `mood` node pay nothing until first use. Straightforward, boring, works everywhere — the safer choice if going the Lottie route at all.

**`@lottiefiles/dotlottie-react`** (v0.19.9, wrapping `@lottiefiles/dotlottie-web` v0.77.1) is a Rust/WASM (ThorVG) renderer built for the newer `.lottie` compressed-bundle format — up to 80% smaller than raw Lottie JSON, and can bundle multiple animations/assets/state-machines in one file. **Critical detail for this app's "must work fully offline" requirement**: by default it lazy-loads its WASM binary from the jsdelivr CDN at runtime — a hard violation of local-first/offline unless corrected. It _is_ fully self-hostable: call `DotLottie.setWasmUrl(wasmUrl)` (or the equivalent export from `dotlottie-react`) with a bundler-emitted local path (Vite's `?url` import) before first render, supported since `dotlottie-react@0.6+` / `dotlottie-wc@0.2+` — the CDN then only acts as a fallback that's never reached. The npm-reported unpacked size of the core package (~9MB) is misleading — that figure includes multiple platform/target WASM builds (WebGL, WebGPU, software); the actual single WASM payload fetched at runtime is on the order of **~150–200KB gzip**, plus the compressed `.lottie` assets themselves.

**Which fits a local-first Vite app better**: if adopting Lottie content at all, `dotlottie-react` (self-hosted WASM + `.lottie` compression) is the technically superior choice for asset weight and offline guarantees, at the cost of an extra one-time setup step (wire `setWasmUrl`, verify no network tab activity in a fully offline browser profile, and convert the Noto JSONs to `.lottie` for the size win). `lottie-react` is simpler and CDN-free by default (no WASM to worry about) but ships the heavier `lottie-web` JS engine and has no compressed-bundle format.

**Lazy-loading strategy** (either runtime): both support code-splitting cleanly — `React.lazy(() => import('lottie-react'))` / dynamic import the dotlottie wrapper, gated behind the same conditional that currently renders `<Face />` only for `mood` nodes. Chat sessions that never render a `mood` widget never pull the renderer or any emoji JSON into their bundle graph, matching the "pay ~0 unless used" goal.

### 3. Fluent Emoji, Rive, and other animated sets

**Microsoft Fluent Emoji (animated)** — MIT-licensed, but the _official_ animated distribution (`microsoft/fluentui-emoji-animated`) ships as **APNG via Git LFS, not Lottie**. Community repos exist that convert a subset to Lottie/JSON, but those aren't officially maintained or guaranteed complete — adopting them means depending on an unofficial, unversioned conversion. Visually, Fluent's style leans even more 3D/glossy than Noto's flat-vector look, which makes the calm-tech clash worse, not better.

**Rive** — the state-machine-driven runtime Duolingo itself uses for its character animation (modular head/body pieces blended in real time, sub-1MB total for 64+ pose combinations). Web runtime is ~200KB gzip including its WASM binary; authored Rive files run 10–15× smaller than equivalent Lottie for the same visual result, and the format _can_ be built theme-aware if the artist designs it that way. The blocker is that **no pre-built library of 8 emotion faces exists in Rive format** — this option means hiring/doing original character animation work in the Rive editor, which is a multi-day-to-multi-week investment, not a drop-in asset swap. Worth revisiting if DorkOS ever wants a signature branded mascot, but out of scope here.

**Discord-style sprite sheets** — Discord's animated emoji use APNG/sprite-sheet frames rather than vector Lottie; this trades smaller per-frame files for coarser fidelity (fixed frame count vs. Lottie's continuous interpolation) and the same full-color theming problem as Noto/Fluent. Not a meaningful upgrade over either option above for this use case.

### 4. Elevate the bespoke motion-SVG faces (recommended)

The repo already has the right foundation: a shared widget-motion vocabulary (`apps/client/src/layers/features/gen-ui/lib/widget-motion.ts` — `WIDGET_SPRING`, `useWidgetMotion()` gated on `useReducedMotion()`), a working blink loop, per-emotion eye/mouth primitives, a heart-eye pulse for `love`, thinking-dots for `thinking`, and a confetti burst + bounce for `celebrating`. The "doesn't look good" complaint is much more likely about **motion character** (linear, small-amplitude, symmetric) than about the SVG-vs-Lottie format itself. The reference points the founder is implicitly reaching for — Duolingo's character work and CSS-only "kawaii" face libraries (e.g. [`kawaii.css`](https://github.com/clzd/kawaii.css), Apache-2.0) — succeed with flat vector shapes, not photoreal/glossy rendering, by leaning hard on classic animation principles:

- **Squash & stretch + anticipation**: a small pre-emptive compress before the "pop" (surprised eyes, celebrating bounce) sells weight that a pure linear scale/opacity never will.
- **Asymmetric timing / secondary motion**: independent, slightly offset timing per eye/eyebrow/mouth element (already partially present via per-dot delays in `ThinkingDots`) reads as more alive than everything moving in lockstep.
- **Organic silhouette**: a very slightly irregular/blobby face outline (a wobbling `border-radius` or a subtly animated circle path) instead of a perfect static circle adds "breathing" life at rest, at near-zero cost.
- **Bespoke per-emotion micro-details**: e.g. a small animated sweat-drop path for `sheepish` (matching the 😅 reference glyph) instead of the current static blush dots; a single falling-tear path for `sad`; a light steam/huff wisp for `determined` instead of static angled brow lines only.

**Cost**: effectively zero new bytes — this is pure component code against primitives already imported (`motion/react`, existing `Face`/`Mouth`/`BlinkEye` structure). **Theming**: full — everything still uses `currentColor` / Tailwind design tokens, so dark mode and any future brand-accent pass fall out for free. **Accessibility**: already correctly wired through `useWidgetMotion()`, which itself wraps `useReducedMotion()` — no new work needed to respect `prefers-reduced-motion`. **Licensing**: none — first-party code. **Effort**: genuinely a one-day scope — it's targeted edits to one ~340-line component file, not a new subsystem.

## Research Gaps & Limitations

- Exact post-gzip size of the 8 target Noto Lottie JSON files was not individually measured (only `1f60a` was fetched and inspected; others were confirmed present in the catalog but not downloaded). Before any future Lottie adoption, pull all 8 candidate files and measure real gzip/brotli sizes.
- The `google/fonts#7011` attribution-clarification GitHub issue does not show a definitive maintainer resolution in the content retrieved — if Noto is ever adopted, get a concrete, current answer on acceptable attribution placement (e.g., a footer/about-page credit vs. per-instance) before shipping, since CC BY 4.0's practical attribution mechanics for a small persistent UI element are genuinely unresolved in the open-source community's own discussion.
- DorkOS has no existing `THIRD_PARTY_LICENSES`/credits surface in the repo (checked — none found); adopting any CC-licensed asset (Noto) would require creating one, which is itself a small scope addition beyond the widget change.
- Did not verify whether `dotlottie-react`'s `setWasmUrl` self-hosting path has any interaction with Vite 6's asset-hashing/CSP setup in this specific repo — flagged as a spike item if the Lottie path is ever pursued.

## Contradictions & Disputes

- Search results were inconsistent about the _exact_ license scope for Noto animated emoji: earlier tool output initially surfaced only the fonts' SIL OFL 1.1 license (from `noto-emoji/LICENSE`) before a targeted follow-up confirmed CC BY 4.0 applies specifically to the _animated artwork_, distinct from the static font glyphs. Treat "CC BY 4.0 for the Lottie/GIF/WebP animated exports, OFL 1.1 for the underlying static font" as the operative summary, but re-verify directly against the repo's current README/LICENSE files before any commercial ship decision — license pages do get restructured.
- `dotlottie-web`'s reported npm "unpacked size" (~9MB) is not representative of runtime cost (it bundles multiple WASM build targets); the credible runtime figure (~150–200KB gzip for the actual WASM payload) comes from secondary community sources (forum posts, migration-issue threads) rather than an official first-party number — worth re-confirming with a real bundle-analyzer run if this path is pursued.

## Recommendation

**Primary — elevate the existing bespoke motion-SVG faces in `MoodNode.tsx`.** Zero licensing risk, zero new asset bytes, full dark-mode/brand theming (everything is already `currentColor`-driven), reduced-motion already correctly wired via `useWidgetMotion()`, and it's a same-file, one-day scope. Apply squash-and-stretch/anticipation to the existing spring transitions, add asymmetric per-element timing, softly organic (non-perfect-circle) idle motion on the face container, and 2–3 new bespoke per-emotion micro-details (sweat drop for `sheepish`, tear for `sad`, steam wisp for `determined`) drawing on the Duolingo/kawaii-CSS animation-principles playbook referenced above.

**Runner-up (future, not one-day scope) — Google Noto Animated Emoji rendered via self-hosted `@lottiefiles/dotlottie-react`.** All 8 emotions map to strong, real glyphs (😊🤔🥳/🎉😅😤😲😢😍), the license (CC BY 4.0 on the animated artwork) is commercially usable with a small attribution addition, and offline viability is fully achievable by wiring `setWasmUrl` to a bundler-local WASM path instead of the CDN default. Hold this in reserve for a future push if DorkOS wants a more universally-recognizable, less bespoke visual language (e.g., for the non-developer persona) — it needs a license-notice surface that doesn't exist yet, a real per-file size audit, an offline-fetch verification pass, and very likely a custom flat/mono re-skin to avoid clashing with the calm, monochrome design system, which together make it a multi-day project rather than a one-day widget polish.
