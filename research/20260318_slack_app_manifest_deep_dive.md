# Slack App Manifest: Format, Fields, Scopes, and URL Creation

## Research Summary

Slack App Manifests are YAML or JSON configuration bundles that fully describe a
Slack app's scopes, event subscriptions, features, and settings. They can be
shared as URL-encoded query parameters (`manifest_yaml=...`) to create a pre-filled
app creation flow — this is the approach already implemented in DorkOS
(`SLACK_CREATE_APP_URL` in `slack-adapter.ts`). Socket Mode is declared with a
single boolean field (`settings.socket_mode_enabled: true`), but app-level tokens
and signing secrets cannot be embedded in manifests — they are generated by Slack
post-creation. The existing DorkOS manifest is correct and comprehensive; the only
gap is missing `reactions:read` and `mpim:history` event subscriptions.

---

## Key Findings

### 1. Format: YAML and JSON Are Functionally Equivalent

Both YAML and JSON are fully supported. Slack's app creation UI accepts either. The
URL parameter variants are `manifest_yaml=` and `manifest_json=`. YAML is generally
more readable for human-authored manifests; JSON is better for programmatic
generation. Both must be URL-encoded when embedded in a URL. The manifest schema is
versioned via a `_metadata` block (v1 is current for standard Slack apps; v2 is
for Slack Workflow Automation apps only).

### 2. Manifest Schema Structure (v1)

Top-level sections:

| Section | Required? | Purpose |
|---|---|---|
| `display_information` | **Required** | App name, description, background color |
| `features` | Optional | Bot user, app home, slash commands, shortcuts |
| `oauth_config` | Optional | Bot/user OAuth scopes and redirect URLs |
| `settings` | Optional | Event subscriptions, interactivity, Socket Mode |
| `functions` | Optional | Custom workflow steps (v2 only) |
| `workflows` | Optional | Workflow definitions (v2 only) |
| `datastores` | Optional | Data persistence (v2 only) |
| `outgoing_domains` | Optional | Egress domain allowlist |
| `_metadata` | Optional | Schema version declaration |

Within `display_information`, only `name` is required. All other top-level sections
and their children are optional unless their parent section is included.

**Minimum valid manifest:**
```yaml
display_information:
  name: My App
```

### 3. URL Parameter Approach for One-Click App Creation

The documented URL pattern for pre-filling app creation is:

```
https://api.slack.com/apps?new_app=1&manifest_yaml=<URL-encoded-YAML>
https://api.slack.com/apps?new_app=1&manifest_json=<URL-encoded-JSON>
```

When a user navigates to this URL:
1. Slack drops them directly into the "Create a New App" flow
2. The manifest is pre-loaded into the configuration editor
3. The user selects a target workspace
4. They review the configuration and click **Create**

Slack's official documentation confirms this approach and calls it "sharing your
manifest as a link."

**URL encoding requirement:** The YAML or JSON must be URL-encoded before embedding.
DorkOS already does this correctly with `encodeURIComponent(SLACK_APP_MANIFEST_YAML)`.

### 4. URL Length: The Practical Limit

Slack's manifest has a documented maximum content length of **4,000 characters**
(min 174). URL-encoding expands size — spaces become `%20`, colons become `%3A`,
newlines become `%0A`, etc. Rough expansion factor: ~1.5–2x.

A 4,000-char manifest becomes ~6,000–8,000 chars URL-encoded. Adding the base URL
(`https://api.slack.com/apps?new_app=1&manifest_yaml=` = 50 chars), the total
approaches 8,050 characters.

**Browser/server limits by environment:**
- Chrome/Firefox: Support up to ~2MB in practice; the `GET` URL limit is imposed by
  the server, not the browser.
- `api.slack.com` server: No documented limit was found. Slack's own documentation
  does not warn about length limits for this parameter.
- Practical limit: The 4,000-char manifest cap is the effective ceiling. A typical
  DorkOS manifest with 12 bot scopes and 5 event subscriptions is approximately
  500–700 characters of YAML, expanding to roughly 800–1,200 chars URL-encoded.
  This is well within any practical limit.

**The DorkOS manifest is short enough that URL length is not a concern.**

### 5. Socket Mode in the Manifest

Socket Mode is declared with a single field in the `settings` section:

```yaml
settings:
  socket_mode_enabled: true
```

That is the **only** manifest field needed to enable Socket Mode. There is no
manifest field for:
- `connections:write` scope (this is an app-level token scope, not an OAuth scope)
- App-level token generation (tokens are created in the Slack UI post-app-creation,
  under **Basic Information → App-Level Tokens**)
- Signing secrets (generated automatically by Slack, not configurable)

When `socket_mode_enabled: true` is set, Slack's UI will prompt users to create an
app-level token with `connections:write` during setup. The manifest cannot
pre-provision this token — users must generate it manually.

**Implication for DorkOS setup UX:** The manifest URL gets users 80% of the way
there, but the `SLACK_APP_TOKEN` (the `xapp-...` token) still requires a manual
step in the Slack UI after app creation.

### 6. App-Level Tokens and Signing Secrets Are Not in Manifests

Confirmed: **No manifest fields exist for:**

| Credential | How Generated |
|---|---|
| Signing Secret | Auto-generated by Slack on app creation (Basic Information → App Credentials) |
| Bot Token (`xoxb-...`) | Issued after OAuth install to a workspace (OAuth & Permissions) |
| App-Level Token (`xapp-...`) | Manually created in Basic Information → App-Level Tokens with `connections:write` |

The manifest `oauth_config.scopes.bot` array controls what OAuth scopes the bot
token will have. The `connections:write` scope required by Socket Mode is an
app-level scope, declared in a separate token section, not in `oauth_config`.

### 7. Recommended Scopes for DorkOS Relay Slack Bot

All scopes should be under `oauth_config.scopes.bot`. **Do not add user-level
scopes** — the "Agents & AI Apps" feature in Slack silently adds these and causes
`invalid_scope` install failures on most workspace plans.

| Scope | Purpose |
|---|---|
| `channels:history` | Read messages in public channels the bot is in |
| `channels:read` | List and get info on public channels |
| `chat:write` | Send messages as the bot |
| `groups:history` | Read messages in private channels the bot is in |
| `groups:read` | List and get info on private channels |
| `im:history` | Read direct messages to the bot |
| `im:read` | Get info on DM channels |
| `im:write` | Open/create DM conversations |
| `mpim:history` | Read group DM messages |
| `app_mentions:read` | Receive events when the bot is @-mentioned |
| `users:read` | Look up user names (for display/routing) |
| `reactions:write` | Add and remove emoji reactions (requires both write+read for removal) |
| `reactions:read` | Read emoji reactions; required to see `reaction_removed` events |

**Note on `reactions:read`:** This scope is listed as missing from the current
DorkOS manifest (`slack-adapter.ts` line 38–66 shows only `reactions:write` in the
bot scopes). Both scopes are needed if the adapter needs to read/remove reactions.

**The current manifest also omits `mpim:history`** from event subscriptions
(present in scopes but not in `bot_events`). This means group DM messages won't
trigger events even though the scope allows reading them.

### 8. Event Subscriptions

Event subscriptions live under `settings.event_subscriptions.bot_events`. For a
DorkOS relay bot, the recommended events are:

```yaml
settings:
  event_subscriptions:
    bot_events:
      - message.channels   # messages in public channels
      - message.groups     # messages in private channels
      - message.im         # direct messages to bot
      - message.mpim       # group direct messages
      - app_mention        # @-mentions of the bot
```

`message.mpim` is the event that fires when a group DM message is sent. It requires
`mpim:history` scope, which is already in the DorkOS manifest scopes list. Adding it
to `bot_events` closes the gap.

For `request_url`: this field is **not needed** when `socket_mode_enabled: true`.
Socket Mode replaces the request URL entirely.

## Complete Recommended Manifest for DorkOS Relay (YAML)

```yaml
display_information:
  name: DorkOS Relay
settings:
  socket_mode_enabled: true
  event_subscriptions:
    bot_events:
      - message.channels
      - message.groups
      - message.im
      - message.mpim
      - app_mention
features:
  bot_user:
    display_name: DorkOS Relay
    always_online: false
oauth_config:
  scopes:
    bot:
      - channels:history
      - channels:read
      - chat:write
      - groups:history
      - groups:read
      - im:history
      - im:read
      - im:write
      - mpim:history
      - app_mentions:read
      - users:read
      - reactions:write
      - reactions:read
```

**Differences from current manifest in `slack-adapter.ts`:**
1. Added `reactions:read` to bot scopes (was missing; only `reactions:write` was present)
2. Added `message.mpim` to `bot_events` (was missing; `mpim:history` scope existed but no matching event)

## What the Manifest Cannot Do

| Capability | Manifest Support |
|---|---|
| Pre-generate signing secret | No — Slack generates this automatically |
| Pre-generate bot token | No — issued after OAuth install to workspace |
| Pre-generate app-level token | No — user creates manually in Basic Information |
| Configure `connections:write` | No — app-level token scope, not an OAuth scope |
| Enable token rotation | Yes — `settings.token_rotation_enabled: true` |
| Specify org deploy | Yes — `settings.org_deploy_enabled: true/false` |
| Restrict IP ranges | Yes — `settings.allowed_ip_address_ranges` |

## How Users Paste/Upload a Manifest in the Slack UI

**Method 1: URL (one-click, recommended for DorkOS)**
- Navigate to the `SLACK_CREATE_APP_URL` link
- Slack pre-fills the manifest editor
- User selects workspace → clicks Create
- Remaining steps: generate app-level token, install app to workspace, copy credentials

**Method 2: Paste in UI**
1. Go to [api.slack.com/apps](https://api.slack.com/apps) → Create New App
2. Choose "From an app manifest"
3. Select workspace
4. Paste YAML or JSON into the editor
5. Click Next → review summary → click Create

**Method 3: apps.manifest.create API**
- POST to `apps.manifest.create` with a JSON manifest body
- Returns `app_id` and credential placeholders
- Useful for automated setup tooling, not for end users

## Research Gaps

- Whether Slack validates the `manifest_yaml` URL server-side before the user
  navigates the app creation flow (e.g., if the YAML is malformed, does the URL
  fail silently or show an error?) — not tested.
- Whether `always_online: false` in the bot_user section has any meaningful effect
  on Socket Mode apps (docs are sparse on this interaction).
- The exact Slack server-side URL length limit for the `manifest_yaml` parameter
  is undocumented; the 4,000-char manifest content limit is the practical ceiling.
